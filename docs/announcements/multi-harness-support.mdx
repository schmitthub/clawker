The multi-harness release changes how images are built, how harnesses authenticate, and how project config is shaped. Most config changes migrate automatically: the first time the new CLI reads your files it rewrites them and prints a notice for every key it moved or removed — you don't need to edit anything it mentions. The checklist below covers the parts that can't be automated.

## Upgrade checklist

<Steps>
<Step title="Update clawker">

```bash
brew upgrade schmitthub/tap/clawker
```

Or re-run the [install script](/installation#install-script). Watch stderr on your first command — the one-shot migration notices print there.
</Step>

<Step title="Rebuild every project image">

```bash
cd your-project
clawker build --no-cache
```

The shipped harness bundles (`claude`, `codex`) and stack definitions (`go`, `node`, `python`, `rust`) resolve straight from the clawker binary — no registration and no config-directory copies. The build tags the image per harness (`clawker-<project>:<harness>` plus a `:default` alias for the default harness).

`--no-cache` matters: images now bake a harness seed directory, and cached layers from a pre-upgrade build skip that step silently — the container comes up without its harness seeds and nothing errors.
</Step>

<Step title="Recreate containers with fresh volumes">

```bash
clawker rm --volumes --agent dev
clawker run -it --agent dev @
```

Old config volumes carry pre-upgrade state. If you reuse one, the container's first-boot marker is missing from its new location, so the control plane re-runs seed apply and your `post_init` hook once against the existing volume — seeds skip files that already exist, but a non-idempotent `post_init` re-fires. Fresh volumes avoid all of it.

Deleting a harness config volume also deletes the harness login stored in it — expect to authenticate again on first run (see [auth changes](#harness-auth-happens-in-container) below).
</Step>

<Step title="Restart the control plane">

```bash
clawker controlplane down
clawker controlplane up
```

Routing the harness command into new containers requires the new control-plane binary; an old control plane leaves newly created containers without their harness CMD.
</Step>

<Step title="Re-initialize the monitoring stack (if you run it)">

With the control plane down, cycle the stack:

```bash
clawker monitor down --volumes
clawker monitor init --force
clawker monitor up
clawker controlplane up
```

A pre-upgrade monitoring stack can leave the new control plane's network logger unable to reach the collector (TLS verification failure); the full cycle regenerates the stack configuration. `--volumes` drops the OpenSearch data volume, so telemetry history resets.
</Step>
</Steps>

## What migrates automatically

Migrations run once per config file the first time the new CLI loads it — the project `.clawker.yaml`, a `.clawker.local.yaml` overlay, and the global `~/.config/clawker/clawker.yaml` are each rewritten independently, with a stderr notice naming every key and the value it carried:

- `build.image`, `build.dockerfile`, `build.context` — removed. Images build from clawker's pinned base; see [build changes](#custom-base-images-and-dockerfiles-are-unsupported).
- `agent.claude_code.use_host_auth` — removed. Host credentials are no longer copied; see [auth changes](#harness-auth-happens-in-container).
- `agent.claude_code` — moved field-for-field to `harnesses.claude`. If a `harnesses.claude` entry already exists, the legacy block is dropped with a warning instead (the map entry already overrides it).

`build.inject.after_claude_install` is not rewritten: it still works as a deprecated alias of `after_harness_install`. Rename it when convenient.

## Behavior changes

### Harness auth happens in-container

Host Claude Code (or any harness) credentials are no longer copied into containers. Instead you authenticate once inside the container on first run — browser OAuth flows are proxied to your host browser automatically — and the login persists in the harness config volume across restarts and recreates that reuse the volume. Managed configuration (settings, plugins, skills) is still staged from the host at container create. See [Credential Forwarding](/credentials#harness-authentication).

### Custom base images and Dockerfiles are unsupported

`build.image`, `build.dockerfile`, and `build.context` are gone. Every image now builds from clawker's pinned Debian substrate, so a config that pointed at a custom base image is left with nothing installing its runtime once those keys are stripped. The migration removes the keys but can't translate what they carried — walk your old base image and re-express each piece through the `build` block.

<Steps>
<Step title="Replace the base image's runtime with a stack">

Start by listing what clawker can install for you:

```bash
clawker stack list
```

This shows the shipped `go`, `node`, `python`, and `rust` stacks plus any your project has registered. Declare the runtimes your old base image carried under `build.stacks` — they render into the shared base image before your build instructions run, so later steps can rely on them:

```yaml
build:
  stacks:
    - node
    - python
```

If your base provided a runtime clawker doesn't ship, author a stack definition and register it, then declare it by name:

```bash
clawker stack register ./stacks/<name>
```

See [Stacks](/stacks) for the definition format and the registry lookup chain.
</Step>

<Step title="Pin interior versions with build args">

A declared stack floats to a current release by default. To reproduce a base image that pinned a version (`node:20-bookworm`, a fixed Go line), pass the stack's version arg at build time:

```bash
clawker build --build-arg NODE_VERSION=20
clawker build --build-arg GO_VERSION=1.24
```

`NODE_VERSION` and `GO_VERSION` name the runtime *line*, not an exact patch — the latest patch on that line resolves per build, so rebuilds float onto security fixes. A build arg that a base-declared stack defines folds into the base freshness hash, so changing it rebuilds the base image.
</Step>

<Step title="Move system packages to build.packages">

apt packages your old Dockerfile installed with `apt-get install` move to `build.packages`. They install into the same apt layer as clawker's tooling floor:

```yaml
build:
  packages:
    - ripgrep
    - libpq-dev
```
</Step>

<Step title="Move everything else to instructions and inject">

Remaining Dockerfile directives map to typed [build instructions](/custom-images#build-instructions): `ENV` becomes `instructions.env`, `COPY` becomes `instructions.copy`, a root-scope `RUN` becomes `instructions.root_run`, and a user-scope `RUN` becomes `instructions.user_run`. Anything with no typed equivalent goes into a raw [injection point](/custom-images#injection-points) — `inject.after_from`, `inject.after_packages`, and the rest.

A step that belongs to only one harness's image — not the shared base — goes in that harness's [build overlay](/custom-images#per-harness-overlays) under `build.harnesses.<name>`, which layers extra stacks, packages, or inject lines onto a single harness without touching the base or the bundle.
</Step>
</Steps>

#### Worked example

**Before** — a custom base image supplied Node and you added one package on top:

```yaml
build:
  image: node:20-bookworm
  packages:
    - ripgrep
```

**After** — declare the `node` stack, keep the package, and pin the line with a build arg:

```yaml
build:
  stacks:
    - node
  packages:
    - ripgrep
```

```bash
clawker build --no-cache --build-arg NODE_VERSION=20
```

See [Image Customization](/custom-images) for the full build block and [Harnesses](/harnesses) for harness-scoped builds.

### Image tags are harness-keyed

Builds produce `clawker-<project>:<harness>` and no longer produce `:latest`. A leftover `:latest` image still resolves as a legacy fallback (with a warning telling you to rebuild). After rebuilding and recreating containers, remove it:

```bash
clawker rmi clawker-<project>:latest
# or sweep all unused clawker images
clawker image prune --all
```

## Review hooks across all config layers

The migration moves the `agent.claude_code` block, but it can't know that a generic `agent.post_init` or `agent.pre_run` contains harness-specific commands. `agent.*` hooks and env apply to **every** harness — a `claude mcp add` line in `agent.post_init` fails inside a codex container. Move harness-specific commands to `harnesses.claude.post_init` / `harnesses.claude.pre_run`.

Check every layer — `.clawker.yaml`, `.clawker.local.yaml`, and the global `~/.config/clawker/clawker.yaml` merge independently, and a forgotten global-layer hook keeps injecting claude commands into every container. Hooks are baked in at container create, so recreate containers after moving them.
