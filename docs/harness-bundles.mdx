A **harness** is the coding-agent CLI a container runs — Claude Code, Codex, or one you bring yourself. Clawker is harness-agnostic: nothing about a harness is hardcoded. Everything a harness needs — how to install it, where it keeps state, what network floor it requires, which files to seed on first boot — lives in a **bundle**, a directory Clawker reads at build time.

Two bundles ship in the binary today, `claude` and `codex`, and resolve without any registration. Authoring a harness for an agent Clawker doesn't ship is the same shape: write a bundle directory, then register its path per-project with `clawker harness register`.

<Note>
This page is about **authoring and selecting harnesses**. For how the two-stage image is built around a bundle (the substrate base, packages, injection points), see [Image Customization](/custom-images). For how a bundle's egress floor composes with your firewall rules, see [Egress Firewall](/firewall).
</Note>

## What a bundle is

A harness bundle is a directory with three parts:

<Card title="harness.yaml" icon="file-lines">
The **manifest** — the data Clawker's engine consumes outside template rendering: the version resolver, declared stacks, persisted volumes, first-boot config seeds, create-time host→container staging, and the egress floor.
</Card>

<Card title="Dockerfile.harness.tmpl" icon="docker">
The **build fragment** — a set of `{{define}}` blocks that fill named slots in Clawker's master harness Dockerfile. This is where the harness CLI is installed and its `CMD` is set. The master owns instruction ordering and cache architecture; a bundle can only fill the declared slots.
</Card>

<Card title="assets/ (optional)" icon="folder">
Every file the bundle contributes to the Docker build context — config seeds, scripts, prompt files. The whole tree is staged verbatim under an `assets/` prefix; the template's `COPY` instructions and the manifest's `seeds[].file` entries reference `assets/`-relative paths.
</Card>

## Shipped bundles

The `claude` and `codex` bundles are embedded in the Clawker binary and load straight from it — there is no materialization step and no config-directory copies. To customize how a shipped harness is built, author your own bundle and register it under the same name (it shadows the shipped one, reported in the build output); see [Overriding a shipped harness](#overriding-a-shipped-harness).

## The registry and lineage lookup

Registration makes a bundle **available**; selecting it at build/run time renders it. Register a harness in your project's `clawker.yaml` under the top-level `harnesses:` map — name → path:

```yaml
# clawker.yaml
harnesses:
  codex:
    path: ./tools/codex-bundle
```

The [`clawker harness register`](/cli-reference/clawker_harness_register) command writes these entries for you. Paths are **relative to the project root** or **absolute** — no `~` or `$VAR` expansion.

A harness name resolves through a **per-lineage lookup chain** — the project `harnesses:` registry, then the shipped embedded bundles as the floor. The closest layer that defines the name wins **wholesale**. A project registration under a shipped name (`claude`, `codex`) **shadows** it, and the build output prints a provenance line so the substitution is never silent:

```
harness codex ← ./tools/codex-bundle (project registry) shadows built
```

<Note>
The `harnesses:` map does double duty: an entry's `path:` registers the bundle, and the same `harnesses.<name>` entry also holds that harness's [per-harness init config](/configuration) (`env`, `post_init`, `pre_run`, managed-config strategy). An entry with init config but no `path:` is not a registration — it configures a harness that resolves elsewhere.
</Note>

<Note>
Registration writes to the project's `clawker.yaml`, so run it from inside an initialized project (`clawker init`). Run from an unregistered directory, the entry lands in your user-level `clawker.yaml` instead — the command prints the file it wrote to.
</Note>

The registry key **is** the image tag, so it follows the shared [naming rule](/stacks#naming): lowercase kebab-case, up to 32 characters (`^[a-z0-9]([a-z0-9-]*[a-z0-9])?$`), directory name unless `--name` overrides. The names `default`, `latest`, and `base` are reserved tag aliases and rejected.

### Registering, listing, and removing

```bash
# Register a bundle directory (name = directory base name); bundled stacks are reported
clawker harness register ./tools/codex-bundle --name codex

# List project-registered and built-in harnesses (SHADOWS flags overrides)
clawker harness list

# Remove a project registration (built-in harnesses can only be shadowed, not removed)
clawker harness remove codex
```

`clawker harness remove` drops the registration path; if the same entry also carries per-harness init config, only the `path` is removed and the init config stays in place.

### Selecting a harness

A build produces one harness image per harness you build; the resulting tag is the harness name. The `@` shortcut in run/create commands resolves against these tags:

```bash
clawker build              # Builds the default harness → clawker-myapp:<harness> + :default alias
clawker build -t codex     # Builds the codex harness  → clawker-myapp:codex

clawker run -it --agent dev @          # Resolves the :default alias
clawker run -it --agent dev @:codex    # Selects the codex harness image
```

Images and containers carry the `dev.clawker.harness` label identifying which harness they were built for (authoritative for filtering).

## Manifest reference (`harness.yaml`)

Every field is validated at load time — an invalid manifest fails the build with a named error, never a silent skip.

### `version`

Declares how the harness version rendered into the build template (`{{.HarnessVersion}}`) is resolved:

```yaml
version:
  resolver: npm                        # npm | github-release | none
  package: "@anthropic-ai/claude-code"
```

| `resolver` | Behavior |
|------------|----------|
| `npm` | Resolves the package's `latest` dist-tag from the npm registry. `package` is the npm package name. |
| `github-release` | Resolves the repo's latest release tag via the GitHub API. `package` is `owner/repo`. `tag_prefix` (optional) is stripped to get the bare version — e.g. `tag_prefix: rust-v` turns tag `rust-v0.50.0` into `0.50.0`. |
| `none` | Renders the floating default (`latest`). No network call. |

Codex uses the GitHub-release resolver against the same tags its own installer downloads:

```yaml
version:
  resolver: github-release
  package: openai/codex
  tag_prefix: rust-v
```

Resolution runs once per build at the command layer. On failure Clawker warns and falls back to the floating default, so a transient registry outage still produces a usable image (the install step downloads whatever "latest" is at build time).

### `stacks`

Names the language stacks the bundle's build blocks require:

```yaml
stacks:
  - node
```

Declared, never installed here — the resolved fragments render in the harness image, always with their lineage-resolved definition (even when your project declared the same name in `build.stacks`; both strata render and the fragment self-guards handle any overlap). Names resolve through the harness lineage: your project's `stacks:` registry, then this bundle's own `stacks/<name>/` subdirectory, then shipped. A bundle embeds a bespoke definition by dropping it under `stacks/<name>/` (the directory name is the stack name) or, to reuse a shared definition, names it without embedding and lets the chain resolve it. See [Stacks](/stacks) for the definition format and lookup chain.

### `volumes`

Each entry declares one persisted directory — a named Docker volume mounted under the container home:

```yaml
volumes:
  - name: config
    path: .codex
```

The volume is named `clawker.<project>.<agent>-<name>` and mounted at `~/<path>`. Clawker assumes nothing about where a harness keeps state — every persisted directory is an explicit declaration.

- `name` must match `[a-zA-Z0-9][a-zA-Z0-9_.-]{0,40}` and be unique. The names `history`, `workspace`, and `clawker` are **reserved** for Clawker infrastructure.
- `path` is container-home-relative and unique across the bundle.

**Every `seeds` and `staging` destination must fall under a declared volume path** — that's the only place state persists. A dest outside every volume is a config error caught at load.

### `seeds`

Files staged into the config volume and applied by Clawker's generic init step on **first boot**:

```yaml
seeds:
  - file: assets/statusline.sh
    dest: .claude/statusline.sh
    apply: copy-if-missing
  - file: assets/claude-config.json
    dest: .claude/.config.json
    apply: copy-if-missing-or-empty
  - file: assets/claude-settings.json
    dest: .claude/settings.json
    apply: json-merge
```

| Field | Meaning |
|-------|---------|
| `file` | Bundle-relative source; **must** live under `assets/` (which is what gets staged into the build context) and exist. |
| `dest` | Container-home-relative target; must fall under a declared volume. |
| `apply` | One of `copy-if-missing`, `copy-if-missing-or-empty`, or `json-merge`. |

Apply strategies:

- **`copy-if-missing`** — write the seed only when the destination does not exist. First boot places it; later boots leave the user's file alone.
- **`copy-if-missing-or-empty`** — also write when the destination exists but is empty.
- **`json-merge`** — deep-merge the seed's JSON into the existing destination JSON, so the seed contributes keys without clobbering user edits.

Because seeds apply into the volume on first boot, a harness comes up preconfigured while a returning container keeps whatever the user changed.

### `staging`

Create-time host→container copies for harness state that lives **outside** the workspace. The workspace itself arrives via bind mount or snapshot and is never staged. Every entry is an explicit, deliberate directive — nothing is copied by naming convention.

```yaml
staging:
  copy:
    - src: ${CLAUDE_CONFIG_DIR:-~/.claude}/settings.json
      dest: .claude/settings.json
      json_keys: [enabledPlugins]
    - src: ${CLAUDE_CONFIG_DIR:-~/.claude}/agents
      dest: .claude/agents
    - src: ${CLAUDE_CONFIG_DIR:-~/.claude}/plugins
      dest: .claude/plugins
      skip: [install-counts-cache.json]
      json_rewrites:
        - { file: known_marketplaces.json, key: installPath, rewrite: prefix-swap }
        - { file: installed_plugins.json, key: projectPath, rewrite: replace-with-workdir }
  mounts:
    - src: ${CLAUDE_CONFIG_DIR:-~/.claude}/projects
      dest: .claude/projects
```

**`copy`** — one-time copies into the volume at create time.

| Field | Meaning |
|-------|---------|
| `src` | Host path or [doublestar](https://github.com/bmatcuk/doublestar) glob. `~`, `$VAR`/`${VAR}`, and shell-style `${VAR:-fallback}` defaults expand before matching. A directory src copies recursively; a glob matching multiple entries lands each under `dest`. Missing sources skip silently. |
| `dest` | Container-home-relative; must fall under a declared volume. |
| `json_keys` | Allowlist: copy only these top-level keys from a JSON file. Requires a single-file `src` (not a glob). |
| `skip` | Filenames to omit when the src is a directory. |
| `json_rewrites` | Per-key path rewrites during copy (see below). |

<Warning>
A src resolving inside the project workspace is **rejected** at stage time — the workspace is mounted, never staged. Point staging at host state that lives outside the repo (a global config dir, a home-relative path).
</Warning>

**`json_rewrites`** rewrite one JSON key's value during the copy, so host-specific paths become valid container paths:

| `rewrite` | Effect |
|-----------|--------|
| `prefix-swap` | Maps the host config-dir tree prefix onto the in-container config-dir tree prefix. |
| `replace-with-workdir` | Substitutes the whole value with the container workspace path. |

**`mounts`** — live host→container bind mounts instead of copies. `src` must be a **literal** path (no globs) and expands the same vocabulary as `copy` src; `dest` is container-home-relative. Use a mount when host and container should share the directory live (e.g. the claude harness mounts `~/.claude/projects` so session history stays shared across container runs).

### `egress`

The harness's required firewall **floor** — the bare-minimum domains the harness needs to function (API, auth, telemetry). Clawker composes this with your project's `security.firewall` rules at create time.

```yaml
egress:
  - dst: api.openai.com
  - dst: auth.openai.com
  - dst: chatgpt.com
    path_rules:
      - path: /backend-api/codex/
        action: allow
  - dst: chatgpt.com
    proto: wss
```

The rule shape (`dst`, `proto`, `port`, `action`, `path_rules`, `path_default`, and each path rule's `path`/`action`/`methods`) is identical to project `security.firewall.rules`. Rather than duplicate the semantics here, see the [Egress Firewall reference](/firewall#configuration) for how domains, path prefixes, protocols, and allowlist vs denylist mode work — and [Harness Floor vs Project Rules](/firewall#harness-floor-vs-project-rules) for how a floor composes with your project rules.

<Note>
Never frame a harness floor as "the system-required rules." Each floor belongs to its harness — the Anthropic domains are the **claude** harness's floor, the OpenAI domains are the **codex** harness's floor. Your custom bundle declares its own.
</Note>

## Template reference (`Dockerfile.harness.tmpl`)

The bundle template fills named block slots in Clawker's master harness Dockerfile. Each `{{define "block_N"}}...{{end}}` body is emitted verbatim at that block's position; you may define any subset of the blocks, and defining anything else is a validation error.

### Block slots

| Block | Scope | Shell | Position |
|-------|-------|-------|----------|
| `block_1` | root | `/bin/sh` | First root step of the harness image (after user creation, which lives in the base). Heavy root installs. |
| `block_2` | root | `/bin/sh` | After base tooling, before the switch to the unprivileged user. |
| `block_3` | user | `/bin/sh` | After the master's static-env section. Environment setup. |
| `block_4` | user | `/bin/zsh` | After `user_run` renders — the version-ARG cache zone. `{{.HarnessVersion}}` is available here. Install the harness CLI here. |
| `block_5` | root | `/bin/zsh` | After the trailing switch back to root, before Clawker's runtime assets. |
| `block_6` | — | — | The final instruction — the `CMD` position. |

Blocks 1–3 run under `/bin/sh`; the master resets `SHELL` to zsh before block_4, so blocks 4–6 run under `/bin/zsh -o pipefail -c`. Stack fragments (when the harness declares them and the project didn't) render **before** block_1 for root-scope fragments and **before** block_3 for user-scope fragments, so every block can rely on them. The manifest `seeds:` stage after block_4. Harness volume directories are pre-created after block_2.

Available in template bodies: `{{.HarnessVersion}}` (resolved version), `${USERNAME}` (the unprivileged container user, re-declared as an ARG after the `FROM`), and `${ZSH_ENV}` (the user's `.zshenv` path, declared before the user-scope stack anchor, so in scope from block_3 on). The reserved names — `Dockerfile` (the composed template itself) plus the project-config inject-point keys (`after_from`, `after_packages`, `after_user_setup`, `after_user_switch`, `after_harness_install` and its deprecated alias `after_claude_install`, `before_entrypoint`) — can never be defined by a bundle.

### Worked example: codex

The codex fragment is small — it sets `CODEX_HOME`, installs the CLI, and sets `CMD`:

```docker
{{define "block_3" -}}
# Tell codex where its config lives — must match the config volume mount.
ENV CODEX_HOME=/home/${USERNAME}/.codex
{{- end}}

{{define "block_4" -}}
ARG CODEX_VERSION={{.HarnessVersion}}
RUN curl -fsSL https://chatgpt.com/codex/install.sh | \
      CODEX_NON_INTERACTIVE=1 CODEX_RELEASE=${CODEX_VERSION} \
      CODEX_HOME=/home/${USERNAME}/.local/share/codex-install sh
{{- end}}

{{define "block_6" -}}
CMD ["codex"]
{{- end}}
```

Note that block_3 points `CODEX_HOME` at `~/.codex` — the declared config volume — while the install (block_4) redirects `CODEX_HOME` to a non-volume path so the image payload isn't shadowed by the runtime volume mount. The version ARG sits directly above its only consumer so a version bump invalidates just the install layer.

### Worked example: claude

The claude fragment uses more of the surface. Block_2 (root scope) writes Claude Code's managed-settings file so any build-time `claude` invocation can find the binary:

```docker
{{define "block_2" -}}
RUN mkdir -p /etc/claude-code && cat > /etc/claude-code/managed-settings.json <<EOF
{
  "env": {
    "PATH": "/home/${USERNAME}/.local/bin:${PATH}"
  }
}
EOF
{{- end}}
```

Block_4 declares the version ARG and installs Claude Code via its official installer — build-only ARG (not ENV), placed directly above the install RUN so a fresh release rolls only the install layer:

```docker
{{define "block_4" -}}
ARG CLAUDE_CODE_VERSION={{.HarnessVersion}}
RUN curl -fsSL "https://claude.ai/install.sh" | bash -s ${CLAUDE_CODE_VERSION}
{{- end}}
```

Block_5 (late root scope) copies a managed agent prompt from the bundle's `assets/`, and block_6 sets the `CMD`:

```docker
{{define "block_5" -}}
COPY assets/clawker-agent-prompt.md /etc/claude-code/CLAUDE.md
{{- end}}

{{define "block_6" -}}
CMD ["claude"]
{{- end}}
```

## Authoring a custom harness

<Steps>
<Step title="Create the bundle directory">
Put it wherever you like; keeping it inside the project (e.g. `./tools/<name>/`) keeps the registry path project-relative. It needs `harness.yaml` and `Dockerfile.harness.tmpl` at minimum, plus an `assets/` tree if you seed or copy files.
</Step>

<Step title="Write the manifest">
Declare the version resolver, any stacks your install steps need, the volume(s) your agent keeps state in, first-boot seeds, create-time staging, and the egress floor. Keep the floor **minimal** — start with just the API and auth endpoints your agent actually needs.
</Step>

<Step title="Write the template">
Fill the block slots. At a minimum you'll want block_4 (install the CLI, using `{{.HarnessVersion}}`) and block_6 (`CMD`). Set your config-dir env var (block_3) to match the config volume path.
</Step>

<Step title="Register it">
Register the bundle directory in your project's `clawker.yaml`:

```bash
clawker harness register ./tools/mytool
```

This writes a `harnesses.mytool.path` entry and reports any stack definitions the bundle embeds under `stacks/`. To register under a different name, pass `--name`.
</Step>

<Step title="Build and run">
```bash
clawker build -t mytool
clawker run -it --agent dev @:mytool
```

Load-time validation runs first: an invalid manifest, a template that defines an unknown block, a seed pointing outside `assets/`, or a staging dest outside every declared volume all fail the build with a named error naming exactly what's wrong. Fix it, rebuild.
</Step>
</Steps>

### Iterating

Edit the bundle directory and rebuild. The base image is unaffected by bundle changes (it's harness-agnostic), so harness-image rebuilds are fast; Docker's layer cache skips unchanged steps. A version bump or an install-step edit invalidates only the install layer and below.

### Overriding a shipped harness

Shipped bundles load from the binary and are never edited in place. To change how a shipped harness is built, author your own bundle and register it under the shipped name:

```bash
clawker harness register ./tools/my-claude --name claude
```

Shadowing a shipped name is an ordinary registration — `--force` is only needed to replace a bundle path your project has already registered. It shadows the shipped `claude` in that project's lineage (the build output reports the shadow). To go back to the shipped bundle, `clawker harness remove claude`.

### Extending a harness without a fork

If you only need to add a stack, a package, or an inject step to a shipped harness — not replace its whole bundle — use the project's [per-harness build overlay](/custom-images#per-harness-overlays) (`build.harnesses.<name>`) instead. It layers onto the bundle's own build without touching it.

### Designing the egress floor

Keep the floor as small as the harness needs to work, and **scope user-generated-content sinks by path**. Both shipped bundles model this:

- The **codex** bundle allows `chatgpt.com` only under the codex backend prefixes (`/backend-api/codex/`, `/backend-api/wham/usage`, and a few others), with no `path_default` — putting the host in allowlist mode so every other path is denied. `chatgpt.com` also hosts publicly readable shared conversations, an exfiltration sink if the whole domain were open.
- The **claude** bundle allows `.claude.ai` broadly (OAuth and login live under `/` and `/login`) but adds `/public/` and `/share/` **deny** path rules so an injected prompt can't pivot the agent into fetching attacker-authored UGC from a trusted origin.

Widen a floor only from **observed** blocked traffic, never preemptively. See the [firewall rule reference](/firewall#configuration) for path-rule semantics.

## Authentication

Harness bundles stage **managed config, never credentials**. Host credentials are never copied into a container. The model is: **authenticate once inside the container on first run** — browser OAuth flows are proxied to your host browser automatically — and the token family persists in the harness's config volume across restarts and recreates that reuse the volume.

What the shipped **claude** bundle stages is managed config: an allowlisted `enabledPlugins` key from `settings.json`, `CLAUDE.md`, and the `agents/`, `skills/`, `commands/`, and `plugins/` trees (with host→container path rewrites). None of that is a credential. Git credentials are handled separately and unchanged — SSH/GPG agent socket forwarding plus HTTPS credential proxying from the host. See [Credential Forwarding](/credentials) for the full picture.

## What the engine does not do

A bundle is a declarative manifest plus a template — deliberately a "dumb sandbox." There are no arbitrary hooks: the engine reads the manifest fields and fills the template slots, and that's the whole surface. If you need setup that isn't expressible as a build step, seed, or stage, that's what the project's `agent.post_init` / `agent.pre_run` and per-harness overlays are for (see [Configuration](/configuration)).

The staging filter verbs (`json_keys`, `json_rewrites`, `skip`) are **JSON-only**. There is no equivalent for other formats — you cannot key-filter a TOML or INI file. This is why the codex bundle deliberately does **not** copy `config.toml`: it embeds host-path-keyed state (a projects trust table, MCP server command paths) and TOML has no `json_keys` equivalent, so codex regenerates it on first run rather than importing a host-specific file wholesale.
