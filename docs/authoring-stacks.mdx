This page is for writing your own [stack](/stacks) — the file-backed recipe that installs a language runtime. If you only need to *use* a stack someone else wrote (declaring it, the shipped `go`/`node`/`python`/`rust` definitions, the registry, parameterizing at build time), start with [Stacks](/stacks) and come back here when you want to author or override a definition.

<Note>
For the precise behaviors the build engine promises an author — resolution order, shadowing, validation front doors — see the [Stack Contract](/stack-contract). This page teaches *how* to write a definition; the contract states *what is guaranteed*.
</Note>

## Definition format

A stack definition is a directory with a manifest and at least one Dockerfile fragment:

<Card title="stack.yaml" icon="file-lines">
A one-field manifest: a human-readable `description`. That's the entire schema — see the [`stack.schema.json`](/schemas/stack.schema.json) reference for the exhaustive field list.
</Card>

<Card title="Dockerfile.stack-root.tmpl" icon="user-shield">
The **root-scope** fragment — rendered in a root-USER region of the generated Dockerfile. For system-wide installs (a runtime on `/usr/local`, a system package manager).
</Card>

<Card title="Dockerfile.stack-user.tmpl" icon="user">
The **user-scope** fragment — rendered in the unprivileged-USER region. For per-user installs (a version manager in `$HOME`, a user-owned stack dir).
</Card>

A definition ships **either or both** fragments; at least one is required ([A3](/stack-contract#author-obligations)). One definition can therefore provision a full stack across both scopes — the shipped `node` definition installs a root-global Node LTS in its root fragment and sets up `nvm` per-user in its user fragment.

```yaml
# stack.yaml
description: >-
  The Rust toolchain via rustup (official installer, stable channel) into
  the user's ~/.cargo and ~/.rustup. Skips itself when cargo is already
  present for the user.
```

Fragments are Go templates rendered against the build context, so they can branch on `{{.BuildKitEnabled}}` (to emit `--mount=type=cache` directives) and reference the same build ARGs the surrounding Dockerfile defines (`${USERNAME}`, `${ZSH_ENV}`, `${PATH}`). The engine never inspects what a fragment installs ([E2](/stack-contract#engine-guarantees)) — it checks the template parses and renders it verbatim. Fragments render top-to-bottom in declaration order and are never reordered ([E1](/stack-contract#engine-guarantees)); to have one stack build on another, declare it later.

## Self-guarding convention

Every fragment **guards its own install** — it checks whether the runtime already exists and skips if so. This is what makes a stack safe to declare even when a base image or another layer already provides it: a deliberate user choice is never overwritten, and the "project and harness both declared `node`" case resolves cleanly ([E5](/stack-contract#engine-guarantees) renders both strata; the guard handles the overlap). The `go` root fragment:

```docker
RUN if command -v go >/dev/null 2>&1; then \
      echo "clawker stack go: existing $(go version) — skipping install"; \
    else \
    ... install ... ; \
    fi
```

The `node` fragment guards on a version floor (keeps an existing node at major ≥ `NODE_MIN_MAJOR`, otherwise installs its own on `/usr/local` and wins `PATH`); `nvm`, `python`/`uv`, and `rust` guard on the presence of `~/.nvm`, `python3`/`uv`, and `cargo` respectively. Follow this pattern in your own definitions — probe first, skip with a message, install in the `else`. The engine never judges whether one stack "satisfies" another ([A2](/stack-contract#author-obligations)); satisfaction lives entirely inside the fragment's guard.

## Naming

Stacks and harnesses share one naming rule ([A4](/stack-contract#author-obligations)):

- Lowercase letters, digits, and internal hyphens: `^[a-z0-9]([a-z0-9-]*[a-z0-9])?$` (no leading or trailing hyphen).
- Up to 32 characters.
- **The directory name is the registered name** unless you override it with `--name` at register time.

## Authoring a custom stack

<Steps>
<Step title="Create the definition directory">
Put it inside your project (e.g. `./stacks/<name>/`) so the registry entry stays project-relative, or anywhere on disk. The name must match `^[a-z0-9]([a-z0-9-]*[a-z0-9])?$` (up to 32 chars); the directory base name becomes the registered name unless you pass `--name`.
</Step>

<Step title="Write stack.yaml">
Just a description:

```yaml
description: >-
  Bun (the JavaScript runtime + package manager) installed per-user via the
  official installer. Skips itself when bun is already on PATH.
```
</Step>

<Step title="Write at least one fragment">
Pick the scope. A per-user runtime that installs into `$HOME` goes in `Dockerfile.stack-user.tmpl`; a system-wide install goes in `Dockerfile.stack-root.tmpl`. Guard the install:

```docker
# Dockerfile.stack-user.tmpl — Bun via the official installer, user-owned.
# Guarded: an existing bun install is left untouched.
RUN if command -v bun >/dev/null 2>&1; then \
      echo "clawker stack bun: existing $(bun --version) — skipping install"; \
    else \
      curl -fsSL https://bun.sh/install | bash && \
      echo 'export PATH="$HOME/.bun/bin:$PATH"' >> ${ZSH_ENV} ; \
    fi
```

The `${ZSH_ENV}` ARG is in scope in the user region — appending a `PATH` export there makes the tool resolve in every shell (zsh sources `.zshenv` on every invocation, interactive and not).
</Step>

<Step title="Register it">
```bash
clawker stack register ./stacks/bun
```

This writes a `stacks.bun` entry to your project's `clawker.yaml`.
</Step>

<Step title="Declare and build">
Declare it in a project or a harness:

```yaml
# clawker.yaml
build:
  stacks:
    - bun
```

```bash
clawker build
```
</Step>
</Steps>

## Load-time validation

Validation runs at the write front door — the `clawker stack register` command resolves the path, checks the name, and fully loads the definition before it touches your `clawker.yaml` ([E15](/stack-contract#engine-guarantees)). It checks the name ([A4](/stack-contract#author-obligations)), the manifest, and each fragment: a fragment that's empty or has a template syntax error is a named error ([A1](/stack-contract#author-obligations)), not a silent skip, and a definition with neither fragment is rejected ([A3](/stack-contract#author-obligations)). An unresolvable declared name later fails the build loudly with the register remedy ([E9](/stack-contract#engine-guarantees)) — there is no quiet fall-through.

## Overriding a shipped stack

Register your own definition under a shipped name (`go`, `node`, `python`, `rust`) and it shadows the shipped one wholesale in every lineage that resolves it — the build output reports the shadow ([E8](/stack-contract#engine-guarantees)). Shipped definitions are an immutable floor ([E17](/stack-contract#engine-guarantees)): they are never removed and never edited in place (there is no materialized copy); you override by registering your own:

```bash
clawker stack register ./stacks/node
```

Shadowing a shipped name is an ordinary registration — `--force` is only needed to replace a name your project has already registered. To go back to the shipped definition, `clawker stack remove node`.

<Note>
Parameterizing a stack at build time — declaring a Docker `ARG` and pinning it with `--build-arg` ([A9](/stack-contract#author-obligations)) — is covered on [Stacks](/stacks#parameterizing-a-stack).
</Note>
