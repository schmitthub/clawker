A **toolchain** is a reusable, named language-runtime definition — the file-backed recipe that installs Go, Node.js, Python, Rust, or a runtime you define. Instead of hand-writing install steps in your Dockerfile, you **declare** a toolchain by name and Clawker renders its install fragment into the right place, in the right USER scope, self-guarded so it skips itself when the runtime is already present.

Toolchains are the layer beneath both project and harness images. A project declares them in `build.toolchains`; a harness bundle declares them in its manifest's `toolchains:`. Same definitions, same registry, resolved from one flat namespace.

<Note>
For where toolchains sit in the overall build (the substrate base, packages, injection points), see [Image Customization](/custom-images). This page is the definition-authoring reference.
</Note>

## Declared, never installed

You never write a toolchain's install commands inline. You name it, and Clawker renders the definition's fragments where they belong:

```yaml
# .clawker.yaml — project declaration renders in the shared base image
build:
  toolchains:
    - go
    - python
```

```yaml
# harness.yaml — a harness's own declaration renders in the harness image
toolchains:
  - node
```

**Stage placement follows who declared it, earliest stage wins:**

- A **project** declaration (`build.toolchains`) renders in the shared **base image**, before your `root_run`/`user_run` instructions — so your build commands can rely on the runtime being present.
- A **harness** declaration (manifest `toolchains:`) renders in the **harness image** — *unless* the project already declared the same toolchain, in which case it lives in the base the harness image builds `FROM`, and every harness image gets it for free.

Each definition is **self-sufficient** — there are no dependency chains between toolchains. If a build needs two runtimes, declare both.

## Definition format

A toolchain definition is a directory with a manifest and at least one Dockerfile fragment:

<Card title="toolchain.yaml" icon="file-lines">
A one-field manifest: a human-readable `description`. That's the entire schema.
</Card>

<Card title="Dockerfile.toolchain-root.tmpl" icon="user-shield">
The **root-scope** fragment — rendered in a root-USER region of the generated Dockerfile. For system-wide installs (a runtime on `/usr/local`, a system package manager).
</Card>

<Card title="Dockerfile.toolchain-user.tmpl" icon="user">
The **user-scope** fragment — rendered in the unprivileged-USER region. For per-user installs (a version manager in `$HOME`, a user-owned toolchain dir).
</Card>

A definition ships **either or both** fragments; at least one is required. One definition can therefore provision a full toolchain across both scopes — the shipped `node` definition installs a root-global Node LTS in its root fragment and sets up `nvm` per-user in its user fragment.

```yaml
# toolchain.yaml
description: >-
  The Rust toolchain via rustup (official installer, stable channel) into
  the user's ~/.cargo and ~/.rustup. Skips itself when cargo is already
  present for the user.
```

Fragments are Go templates rendered against the build context, so they can branch on `{{.BuildKitEnabled}}` (to emit `--mount=type=cache` directives) and reference the same build ARGs the surrounding Dockerfile defines (`${USERNAME}`, `${ZSH_ENV}`, `${PATH}`).

### Self-guarding convention

Every fragment **guards its own install** — it checks whether the runtime already exists and skips if so. This is what makes a toolchain safe to declare even when a base image or another layer already provides it: a deliberate user choice is never overwritten. The `go` root fragment:

```docker
RUN if command -v go >/dev/null 2>&1; then \
      echo "clawker toolchain go: existing $(go version) — skipping install"; \
    else \
    ... install ... ; \
    fi
```

The `node` fragment guards on a version floor (keeps an existing node at major ≥ `NODE_MIN_MAJOR`, otherwise installs its own on `/usr/local` and wins `PATH`); `nvm`, `python`/`uv`, and `rust` guard on the presence of `~/.nvm`, `python3`/`uv`, and `cargo` respectively. Follow this pattern in your own definitions — probe first, skip with a message, install in the `else`.

## Shipped definitions

Four toolchains ship in the binary, materialized under your config directory and registered in `settings.yaml`:

| Toolchain | Scope | What it provisions |
|-----------|-------|--------------------|
| `go` | root | The Go toolchain on `/usr/local/go` from the official tarball, sha256-verified against the go.dev download index. Floats to the latest stable unless `GO_VERSION` pins a line. Sets the golang-image `GOPATH`/`PATH` conventions (`/go`, world-writable) so `go install` binaries resolve at runtime for every user. |
| `node` | root + user | Root-global Node LTS on `/usr/local` (GPG-verified prebuilt, floating patch within the `NODE_VERSION` line, `NODE_USE_SYSTEM_CA` trust of the OS CA bundle) **plus** `nvm` as the per-user runtime version switcher. |
| `python` | root | `uv` (Astral's package/project manager) installed system-wide on `/usr/local/bin`, plus a uv-managed CPython installed system-wide with `python3`/`python` on `PATH` for every user and build step. |
| `rust` | user | The Rust toolchain via `rustup` (official installer, stable channel) into the user's `~/.cargo` and `~/.rustup`. |

A few properties worth knowing:

- **`node` and `NODE_USE_SYSTEM_CA`** — the node fragment sets `ENV NODE_USE_SYSTEM_CA=1` globally, so `fetch()` and TLS calls from any node process trust the OS CA bundle — and therefore the [firewall's MITM CA](/firewall#certificate-management) once it's added to the trust store. `NODE_VERSION` (default `24`) names the LTS *line*, not a patch; the latest patch resolves per-build so rebuilds float onto security patches.
- **`python` uses uv-managed CPython** — no system `python3` package; `uv` installs the interpreter and Clawker symlinks `python3`/`python` onto `PATH`. The managed-python directory is world-writable so any user can add versions at runtime.
- **Floating installers** — `nvm`, `uv`, and `rust` install via their canonical `curl | sh` installers, deliberately floated rather than pinned (a frozen installer tag rots into an unpatched script). Each installer verifies the artifacts it downloads.

Because they're materialized as ordinary files, you can open any shipped definition under `<config-dir>/toolchains/<name>/` and edit it — your edits win, and Clawker never overwrites a file you have.

## The registry

Toolchains are registered in `settings.yaml` under `toolchains:`, a map of name → entry:

```yaml
toolchains:
  go:
    path: /home/you/.config/clawker/toolchains/go
  node:
    path: /home/you/.config/clawker/toolchains/node
  python:
    path: /home/you/.config/clawker/toolchains/python
  rust:
    path: /home/you/.config/clawker/toolchains/rust
```

Each entry carries an explicit `path:` to the definition directory (`toolchain.yaml` + at least one fragment). Shipped definitions are materialized copy-if-missing to `<config-dir>/toolchains/<name>/` and their registry entries seeded automatically; you only touch an entry to relocate a definition or register a custom one.

### One flat namespace and collision rules

All toolchain sources — the shipped set, the settings registry, and definitions a harness bundle embeds under its own `toolchains/` subdirectory — share **one flat namespace per build**. A name resolves to exactly one definition. The engine is explicit over clever about collisions:

- A name available from **both** a harness bundle **and** the shipped/registry set is a **collision error**. A bundle-embedded definition may never shadow a shipped or registered one — rename the bundle-embedded definition (bundle authors prefix bespoke definitions). This check also fires when a project declares a name the selected bundle happens to embed, so a bundle can never silently shadow the definition the base actually used.
- A name available only from the bundle resolves to the bundle's definition.
- Otherwise it resolves from the registry, then the shipped set.
- An unknown name is an error naming the known sources.

### Cache implications of editing a definition

A project-declared toolchain's fragments render **into the base Dockerfile**, and the base image's freshness key is a content hash of the rendered base Dockerfile bytes (plus your `instructions.copy` sources). So **editing a declared toolchain definition changes the rendered base and triggers a base rebuild** on the next build — which is what you want: the new install steps take effect. Editing a definition your project doesn't declare has no effect on the base. Use `clawker build --no-cache` to force a full rebuild regardless.

## Authoring a custom toolchain

<Steps>
<Step title="Create the definition directory">
Put it at `<config-dir>/toolchains/<name>/` (the conventional location) or anywhere you like. The name must match `[a-zA-Z0-9][a-zA-Z0-9._-]{0,40}`.
</Step>

<Step title="Write toolchain.yaml">
Just a description:

```yaml
description: >-
  Bun (the JavaScript runtime + package manager) installed per-user via the
  official installer. Skips itself when bun is already on PATH.
```
</Step>

<Step title="Write at least one fragment">
Pick the scope. A per-user runtime that installs into `$HOME` goes in `Dockerfile.toolchain-user.tmpl`; a system-wide install goes in `Dockerfile.toolchain-root.tmpl`. Guard the install:

```docker
# Dockerfile.toolchain-user.tmpl — Bun via the official installer, user-owned.
# Guarded: an existing bun install is left untouched.
RUN if command -v bun >/dev/null 2>&1; then \
      echo "clawker toolchain bun: existing $(bun --version) — skipping install"; \
    else \
      curl -fsSL https://bun.sh/install | bash && \
      echo 'export PATH="$HOME/.bun/bin:$PATH"' >> ${ZSH_ENV} ; \
    fi
```

The `${ZSH_ENV}` ARG is in scope in the user region — appending a `PATH` export there makes the tool resolve in every shell (zsh sources `.zshenv` on every invocation, interactive and not).
</Step>

<Step title="Register it in settings">
Add an entry under `toolchains:` in `settings.yaml`:

```yaml
toolchains:
  bun:
    path: /home/you/.config/clawker/toolchains/bun
```
</Step>

<Step title="Declare and build">
Declare it in a project or a harness:

```yaml
# .clawker.yaml
build:
  toolchains:
    - bun
```

```bash
clawker build
```

Load-time validation checks the name, the manifest, and each fragment (a fragment that's empty or has a template syntax error is a named error, not a silent skip). A definition with neither fragment is rejected.
</Step>
</Steps>

### Overriding a shipped toolchain

Because shipped definitions are just materialized files, "overriding" is editing. Open `<config-dir>/toolchains/node/` (or whichever) and change the fragment — pin a version, add a package, adjust the guard. Your edit persists (Clawker never overwrites an existing file) and, for a project-declared toolchain, the base rebuilds on your next build to pick it up. To relocate a definition entirely, point its registry `path:` at the new directory.
