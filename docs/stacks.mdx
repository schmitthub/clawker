A **stack** is a reusable, named language-runtime definition — the file-backed recipe that installs Go, Node.js, Python, Rust, or a runtime someone defines. Instead of hand-writing install steps in your Dockerfile, you **declare** a stack by name and Clawker renders its install fragment into the right place, in the right USER scope, self-guarded so it skips itself when the runtime is already present.

Stacks are the layer beneath both project and harness images. A project declares them in `build.stacks`; a harness bundle declares them in its manifest's `stacks:`. Same definition format, resolved through a per-lineage lookup chain.

<Note>
For where stacks sit in the overall build (the substrate base, packages, injection points), see [Image Customization](/custom-images). To write or override a stack definition — the manifest format, the self-guard convention, the naming rule — see [Authoring Stacks](/authoring-stacks). This page covers using stacks someone else already wrote.
</Note>

## Declared, never installed

You never write a stack's install commands inline. You name it, and Clawker renders the definition's fragments where they belong:

```yaml
# clawker.yaml — project declaration renders in the shared base image
build:
  stacks:
    - go
    - python
```

```yaml
# harness.yaml — a harness's own declaration renders in the harness image
stacks:
  - node
```

**Stage placement follows who declared it, and the two strata never interact:**

- A **project** declaration (`build.stacks`) renders in the shared **base image**, before your `root_run`/`user_run` instructions — so your build commands can rely on the runtime being present.
- A **harness** declaration (manifest `stacks:`) renders in the **harness image**, *always* with its own lineage-resolved definition — even when the project declared the same name in the base. Both render; nothing is deduplicated across the two strata.

Each definition is **self-sufficient** — there are no dependency chains between stacks. If a build needs two runtimes, declare both. When a runtime is already present (because another stack, the base image, or a farther layer provisioned it), the fragment's self-guard skips its own install — that is how the base-and-harness-both-declare-`node` case stays safe. See the [self-guarding convention](/authoring-stacks#self-guarding-convention) for how a fragment probes and skips.

## Shipped definitions

Four stacks ship in the Clawker binary and resolve without any registration:

| Stack | Scope | What it provisions |
|-----------|-------|--------------------|
| `go` | root | The Go toolchain on `/usr/local/go` from the official tarball, sha256-verified against the go.dev download index. Floats to the latest stable unless `GO_VERSION` pins a line. Sets the golang-image `GOPATH`/`PATH` conventions (`/go`, world-writable) so `go install` binaries resolve at runtime for every user. |
| `node` | root + user | Root-global Node LTS on `/usr/local` (GPG-verified prebuilt, floating patch within the `NODE_VERSION` line, `NODE_USE_SYSTEM_CA` trust of the OS CA bundle) **plus** `nvm` as the per-user runtime version switcher. |
| `python` | root | `uv` (Astral's package/project manager) installed system-wide on `/usr/local/bin`, plus a uv-managed CPython installed system-wide with `python3`/`python` on `PATH` for every user and build step. |
| `rust` | user | The Rust toolchain via `rustup` (official installer, stable channel) into the user's `~/.cargo` and `~/.rustup`. |

A few properties worth knowing:

- **`node` and `NODE_USE_SYSTEM_CA`** — the node fragment sets `ENV NODE_USE_SYSTEM_CA=1` globally, so `fetch()` and TLS calls from any node process trust the OS CA bundle — and therefore the [firewall's MITM CA](/firewall#certificate-management) once it's added to the trust store. `NODE_VERSION` (default `24`) names the LTS *line*, not a patch; the latest patch resolves per-build so rebuilds float onto security patches.
- **`python` uses uv-managed CPython** — no system `python3` package; `uv` installs the interpreter and Clawker symlinks `python3`/`python` onto `PATH`. The managed-python directory is world-writable so any user can add versions at runtime.
- **Floating installers** — `nvm`, `uv`, and `rust` install via their canonical `curl | sh` installers, deliberately floated rather than pinned (a frozen installer tag rots into an unpatched script). Each installer verifies the artifacts it downloads.

## The registry and lineage lookup

Registration makes a definition **available**; declaration renders it. The two are orthogonal.

Register a stack in your project's `clawker.yaml` under the top-level `stacks:` map — name → path:

```yaml
# clawker.yaml
stacks:
  my-rust:
    path: ./stacks/my-rust
```

The [`clawker stack register`](/cli-reference/clawker_stack_register) command writes these entries for you; you rarely edit them by hand. Paths are **relative to the project root** (kept portable within the project) or **absolute** — no `~` or `$VAR` expansion.

A declared name resolves through a **per-lineage lookup chain** — the closest layer that defines the name wins **wholesale** (never merged):

| Where it renders | Lookup chain |
|------------------|--------------|
| Base image (project `build.stacks`) | project `stacks:` registry → shipped |
| Harness image (bundle `stacks:` or `build.harnesses.<name>.stacks`) | project `stacks:` registry → the selected bundle's `stacks/<name>/` dir → shipped |

Shipped definitions are the floor of every chain — they load straight from the binary, with no materialization step and no config-directory copies. A project registration under a shipped name **shadows** it; a bundle-embedded definition shadows shipped in the harness lineage only. Because there is no global namespace, two harness bundles never collide — sibling images don't share a lookup chain.

When a closer layer shadows a farther one, the build output prints a **provenance line** so the substitution is never silent:

```text
stack node ← project (./stacks/node) shadows built
```

An unresolvable name is a hard error naming the `clawker stack register` remedy.

### Registering, listing, and removing

```bash
# Register a definition directory (name = directory base name)
clawker stack register ./stacks/my-rust

# Register under an explicit name
clawker stack register ./vendor/rustup --name rust

# Replace an existing project registration (reports the prior path)
clawker stack register ./stacks/my-rust --force

# List project-registered and built-in stacks (SHADOWS flags overrides)
clawker stack list

# Remove a project registration (built-in stacks can only be shadowed, not removed)
clawker stack remove my-rust
```

<Note>
Registration writes to the project's `clawker.yaml`, so run it from inside an initialized project (`clawker init`). Run from an unregistered directory, the entry lands in your user-level `clawker.yaml` instead — the command prints the file it wrote to, so you always see where it went.
</Note>

Registering your own definition under a shipped name (`go`, `node`, `python`, `rust`) shadows the shipped one; see [overriding a shipped stack](/authoring-stacks#overriding-a-shipped-stack) for the full semantics.

### Cache implications of editing a definition

A project-declared stack's fragments render **into the base Dockerfile**, and the base image's freshness key is a content hash of the rendered base Dockerfile bytes (plus your `instructions.copy` sources). So **editing a declared stack definition changes the rendered base and triggers a base rebuild** on the next build — which is what you want: the new install steps take effect. Editing a definition your project doesn't declare has no effect on the base. Use `clawker build --no-cache` to force a full rebuild regardless.

## Parameterizing a stack

A stack has no version of its own — it is a collection of tools (the `python` stack is `uv` + CPython; `node` is `nvm` + Node). To pin an interior version, the definition's author declares a Docker `ARG` and you override it at build time:

```bash
clawker build --build-arg NODE_VERSION=22
```

The shipped `node`, `go`, and other stacks follow this pattern (`NODE_VERSION`, `GO_VERSION`). This is native Docker — nothing in the config schema. When a `--build-arg` targets an ARG the rendered **base** image declares, Clawker folds the value into the base freshness hash so the base rebuilds when you change it. For variance a single ARG can't express, fork the definition (own every interior choice) or hand-roll via [injection points](/custom-images#injection-points).
