Clawker generates and builds your project's Docker images itself. There is no user-supplied base image and no custom Dockerfile — every image starts from a pinned Debian substrate, and all customization flows through the `build` block of `.clawker.yaml`: system packages, language stacks, typed build instructions, and raw Dockerfile injection points.

## The Two-Stage Build

Each project gets a shared **base image** plus one **harness image** per coding-agent harness you build:

| Image | Tag | Contents |
|-------|-----|----------|
| Base image | `clawker-<project>:base` | Harness-agnostic layers: the substrate, Clawker's tooling floor, your `build.packages`, project-declared stacks, your `instructions` and base-stage `inject` points, the unprivileged container user (`claude`) |
| Harness image | `clawker-<project>:<harness>` | Built `FROM` the base: the harness's own stacks, the bundle's install steps, config seeds, harness-stage `inject` points, and Clawker's runtime assets (`clawkerd`, credential helpers, firewall CA) |

The base image starts from a pinned `debian:bookworm-slim` digest and layers in the development floor every container gets: `git`, `zsh`, `sudo`, `gh`, `curl`, `wget`, `jq`, `fzf`, `make`, `gcc`, editors (`nano`, `vim`), locales, the Docker CLI (for socket-mount workflows), and `git-delta`. Your `build.packages` install into the same apt layer.

The harness image adds whatever the selected harness bundle declares — for example, the `claude` bundle installs Claude Code (plus the `node` stack it runs on), while the `codex` bundle installs the standalone Codex binary — and ends with `ENTRYPOINT ["clawkerd"]` and the harness's `CMD`.

### Caching

The base image is keyed by a content hash of its inputs (the rendered base Dockerfile plus the contents of any `instructions.copy` sources), stamped as an image label. Rebuilding a harness image reuses the existing base whenever that hash is unchanged — editing source files outside your `copy` sources never triggers a base rebuild. Within each stage, Docker's normal layer cache (BuildKit or classic) skips unchanged layers. Use `clawker build --no-cache` to force a full rebuild.

## Building and Tags

```bash
clawker build              # Builds the default harness → clawker-myapp:<harness> + :default alias
clawker build -t codex     # Builds the codex harness → clawker-myapp:codex
```

Tags are harness-keyed: `-t NAME` builds that registered harness, and the resulting tag *is* the harness name. Building the registry-default harness also stamps the `:default` alias.

The `@` shortcut in run/create commands resolves against these tags:

```bash
clawker run -it --agent dev @         # Resolves the :default alias
clawker run -it --agent dev @:codex   # Selects the codex harness image
```

Images and containers carry the `dev.clawker.harness` label identifying which harness they were built for.

Build-time variables can be overridden with `--build-arg` on `clawker build` — for example `--build-arg CLAUDE_CODE_VERSION=2.1.4` pins the claude harness's install to an exact version.

## System Packages

Install additional Debian (apt) packages with the `packages` field:

```yaml
build:
  packages:
    - ripgrep
    - python3
    - libpq-dev
```

## Stacks

Stacks are reusable language-runtime definitions. Declaring one installs it in the shared base image, before your `root_run`/`user_run` instructions execute — so your instructions can rely on it:

```yaml
build:
  stacks:
    - go
    - python
```

Shipped stack definitions:

| Stack | What it installs |
|-----------|------------------|
| `go` | The Go toolchain on `/usr/local/go` (official tarball, checksum-verified) with the standard `GOPATH` conventions |
| `node` | Node.js LTS installed root-global on `/usr/local` (GPG-verified), plus `nvm` as the per-user version switcher |
| `python` | `uv` (Astral's package/project manager) system-wide, plus a uv-managed CPython with `python3`/`python` on `PATH` |
| `rust` | The Rust toolchain via `rustup` (stable channel) into the user's `~/.cargo` and `~/.rustup` |

Stacks are self-guarding — each skips itself when the image already provides the runtime. Harness bundles declare their own stack requirements (for example, the `claude` bundle declares `node`); a harness-declared stack installs in the harness image unless your project already declared it, in which case it lives in the shared base and every harness image gets it for free.

Stack definitions are file-backed (`stack.yaml` plus root- and/or user-scope Dockerfile fragments). Shipped definitions are materialized under your config directory and registered in `settings.yaml` under `stacks:`; you can register custom definitions there too. All sources share one flat namespace — a name collision is an error.

## Build Instructions

The `instructions` block provides type-safe build directives. All of these render in the shared base image.

### env

Environment variables baked into the image:

```yaml
build:
  instructions:
    env:
      NODE_ENV: "development"
      PYTHONPATH: "/workspace/lib"
```

### copy

Copy files from your project into the image with optional `chown`/`chmod`:

```yaml
build:
  instructions:
    copy:
      - src: "./config/eslint.json"
        dest: "/home/claude/.eslintrc.json"
        chown: "claude:claude"
      - src: "./scripts/setup.sh"
        dest: "/usr/local/bin/setup.sh"
        chmod: "755"
```

Copy sources are part of the base image's content hash — editing a copied file triggers a base rebuild; editing other project files does not. `copy` steps render after `user_run` so expensive setup layers stay cached when a copied file changes.

### labels

Add metadata labels to the image:

```yaml
build:
  instructions:
    labels:
      maintainer: "team@example.com"
      version: "1.0"
```

### args

Define build-time arguments (override with `--build-arg` at build time):

```yaml
build:
  instructions:
    args:
      - name: "APP_ENV"
        default: "production"
      - name: "CUSTOM_FLAG"
```

### root_run and user_run

Run commands during the build. `root_run` executes as root, before the container user is created; `user_run` executes as the container user, after the user switch. Both run after any project-declared stacks, so `go`, `node`, `python`, etc. are available:

```yaml
build:
  instructions:
    root_run:
      - "curl -fsSL https://example.com/setup.sh | bash"
    user_run:
      - "pip install --user poetry"
      - "npm install -g typescript"
```

<Note>
  `user_run` commands execute under `/bin/zsh -o pipefail -c`, not `/bin/sh` —
  clawker bakes zsh as the build shell so its shell tooling is available. Two
  consequences for scripts written against POSIX `sh`: zsh does not word-split
  unquoted variable expansions (quote your variables), and `pipefail` is active
  (a failing command anywhere in a pipe fails the build). `root_run` runs earlier
  in the build and is unaffected — it still uses `/bin/sh`.
</Note>

## Injection Points

For advanced use cases, the `inject` field provides raw Dockerfile injection points. Each field accepts an array of raw Dockerfile lines. Four points render in the base image, two in the harness image:

```yaml
build:
  inject:
    after_from:
      - "ARG TARGETARCH"
      - "RUN echo 'Custom base setup'"
    after_packages:
      - "RUN curl -fsSL https://deb.nodesource.com/setup_20.x | bash -"
    after_user_setup:
      - "RUN mkdir -p /home/claude/.config"
    after_user_switch:
      - "COPY --chown=claude:claude ./dotfiles/ /home/claude/"
    after_harness_install:
      - "RUN claude mcp add memory -- npx -y @modelcontextprotocol/server-memory"
    before_entrypoint:
      - "RUN echo 'Final cleanup'"
```

**Injection point order across the generated build:**

*Base image (`clawker-<project>:base`):*

1. `FROM` the pinned substrate
2. `after_from` — apt sources, proxy config, CA certs that package installation depends on
3. Package installation (Clawker's floor + your `build.packages`)
4. `after_packages` — post-install configuration as root
5. Project-declared stacks (root scope), then your `root_run`
6. Container user creation
7. `after_user_setup` — directories, permissions, services (still root)
8. `USER` switch to the unprivileged container user (`claude`)
9. `after_user_switch` — dotfiles, shell config
10. Project-declared stacks (user scope), then your `user_run`, then `copy`

*Harness image (`clawker-<project>:<harness>`, built `FROM` the base):*

11. Harness-declared stacks + the bundle's install steps (e.g. the harness CLI)
12. `after_harness_install` — runs as the container user with the harness CLI available: add MCP servers, install plugins or extensions
13. `before_entrypoint` — final tweaks before Clawker's runtime assets and `ENTRYPOINT`

Because `after_harness_install` and `before_entrypoint` live in the harness image, they run once per harness you build — a step that registers an MCP server with `claude` applies to the claude harness image only.

## Complete Example

```yaml
build:
  packages:
    - ripgrep
    - python3
  stacks:
    - node
  instructions:
    env:
      NODE_ENV: "development"
    copy:
      - src: ".npmrc"
        dest: "/home/claude/.npmrc"
        chown: "claude:claude"
    root_run:
      - "apt-get install -y libsqlite3-dev"
    user_run:
      - "npm install -g typescript ts-node"
  inject:
    after_harness_install:
      - "RUN claude mcp add context7 -- npx -y @upstash/context7-mcp"
```
