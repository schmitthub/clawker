The 0.13 multi-harness release changes how images are built, how harnesses authenticate, and how project config is shaped. Most config changes migrate automatically: the first time the new CLI reads your files it rewrites them and prints a notice for every key it moved or removed â€” you don't need to edit anything it mentions. The checklist below covers the parts that can't be automated.

## Upgrade checklist

<Steps>
<Step title="Update clawker">

```bash
brew upgrade schmitthub/tap/clawker
```

Or re-run the [install script](/installation#install-script). Watch stderr on your first command â€” the one-shot migration notices print there.
</Step>

<Step title="Rebuild every project image">

```bash
cd your-project
clawker build --no-cache
```

The shipped harness bundles (`claude`, `codex`) and stack definitions (`go`, `node`, `python`, `rust`) resolve straight from the clawker binary â€” no registration and no config-directory copies. The build tags the image per harness (`clawker-<project>:<harness>` plus a `:default` alias for the default harness).

`--no-cache` matters: images now bake a harness seed directory, and cached layers from a pre-upgrade build skip that step silently â€” the container comes up without its harness seeds and nothing errors.
</Step>

<Step title="Recreate containers with fresh volumes">

```bash
clawker rm --volumes --agent dev
clawker run -it --agent dev @
```

Old config volumes carry pre-upgrade state. If you reuse one, the container's first-boot marker is missing from its new location, so the control plane re-runs seed apply and your `post_init` hook once against the existing volume â€” seeds skip files that already exist, but a non-idempotent `post_init` re-fires. Fresh volumes avoid all of it.

Deleting a harness config volume also deletes the harness login stored in it â€” expect to authenticate again on first run (see [auth changes](#harness-auth-happens-in-container) below).
</Step>

<Step title="Restart the control plane">

```bash
clawker controlplane down
clawker controlplane up
```

Routing the harness command into new containers requires the new control-plane binary; an old control plane leaves newly created containers without their harness CMD.
</Step>

<Step title="Re-initialize the monitoring stack (if you run it)">

With the control plane down, cycle the stack:

```bash
clawker monitor down --volumes
clawker monitor init --force
clawker monitor up
clawker controlplane up
```

A pre-upgrade monitoring stack can leave the new control plane's network logger unable to reach the collector (TLS verification failure); the full cycle regenerates the stack configuration. `--volumes` drops the OpenSearch data volume, so telemetry history resets.
</Step>
</Steps>

## What migrates automatically

Migrations run once per config file the first time the new CLI loads it â€” the project `.clawker.yaml`, a `.clawker.local.yaml` overlay, and the global `~/.config/clawker/clawker.yaml` are each rewritten independently, with a stderr notice naming every key and the value it carried:

- `build.image`, `build.dockerfile`, `build.context` â€” removed. Images build from clawker's pinned base; see [build changes](#custom-base-images-and-dockerfiles-are-unsupported).
- `agent.claude_code.use_host_auth` â€” removed. Host credentials are no longer copied; see [auth changes](#harness-auth-happens-in-container).
- `agent.claude_code` â€” moved field-for-field to `harnesses.claude`. If a `harnesses.claude` entry already exists, the legacy block is dropped with a warning instead (the map entry already overrides it).
- Settings-side `stacks:` / `harnesses:` registries â€” stripped from `settings.yaml` on load with a notice. Registration is now per-project in `clawker.yaml`; shipped harnesses and stacks resolve from the binary and need no registry entry. Register your own with `clawker stack register` / `clawker harness register` (see [Stacks](/stacks) and [Harness Bundles](/harness-bundles)).

`build.inject.after_claude_install` is not rewritten: it still works as a deprecated alias of `after_harness_install`. Rename it when convenient.

## Behavior changes

### Harness auth happens in-container

Host Claude Code (or any harness) credentials are no longer copied into containers. Instead you authenticate once inside the container on first run â€” browser OAuth flows are proxied to your host browser automatically â€” and the login persists in the harness config volume across restarts and recreates that reuse the volume. Managed configuration (settings, plugins, skills) is still staged from the host at container create. See [Credential Forwarding](/credentials#harness-authentication).

### Custom base images and Dockerfiles are unsupported

Every image builds from clawker's pinned Debian base. Declare language stacks with `build.stacks` (`go`, `node`, `python`, `rust`, or your own definitions), and customize the image with `build.packages`, `build.instructions`, and `build.inject`. See [Custom Images](/custom-images) and [Stacks](/stacks).

### Image tags are harness-keyed

Builds produce `clawker-<project>:<harness>` and no longer produce `:latest`. A leftover `:latest` image still resolves as a legacy fallback (with a warning telling you to rebuild). After rebuilding and recreating containers, remove it:

```bash
clawker rmi clawker-<project>:latest
# or sweep all unused clawker images
clawker image prune --all
```

## Review hooks across all config layers

The migration moves the `agent.claude_code` block, but it can't know that a generic `agent.post_init` or `agent.pre_run` contains harness-specific commands. `agent.*` hooks and env apply to **every** harness â€” a `claude mcp add` line in `agent.post_init` fails inside a codex container. Move harness-specific commands to `harnesses.claude.post_init` / `harnesses.claude.pre_run`.

Check every layer â€” `.clawker.yaml`, `.clawker.local.yaml`, and the global `~/.config/clawker/clawker.yaml` merge independently, and a forgotten global-layer hook keeps injecting claude commands into every container. Hooks are baked in at container create, so recreate containers after moving them.

## Alpha builds: drop stale config-dir copies

If you ran earlier pre-release builds of this branch, they may have left harness and stack copies under `~/.config/clawker/harnesses/` and `~/.config/clawker/stacks/`. Shipped harnesses and stacks now resolve from the binary, so those directories are no longer read â€” delete them:

```bash
rm -rf ~/.config/clawker/harnesses ~/.config/clawker/stacks
```

Any custom bundles or stacks you want to keep, register per-project with `clawker harness register` / `clawker stack register` pointing at wherever you keep them on disk.
