## 3. ATTACK CHAIN ‚Äî OBJECTIVE 1: FIREWALL BYPASS

### Seam Exploited: NET_ADMIN capability grants iptables control to container process

**Attack steps:**

1. Container has `NET_ADMIN` and `NET_RAW` capabilities (default Clawker config ‚Äî required for firewall)
2. From inside container: `iptables -F -t nat` ‚Äî flushes all NAT rules (exit code 0)
3. After flush: `curl http://example.com` ‚Äî HTTP 200 from `104.18.26.120` (unauthorized domain)
4. After flush: `curl https://httpbin.org/get` ‚Äî HTTP 200 from `52.71.170.232` (unauthorized domain)
5. DNS resolution works: `dig +short google.com` ‚Üí `142.250.73.142`

**Proof:**
```
$ docker exec clawker-redteam-live iptables -F -t nat
(exit 0)
$ docker exec clawker-redteam-live curl -s -o /dev/null -w "HTTP %{http_code}" http://example.com
HTTP 200
$ docker exec clawker-redteam-live curl -s -o /dev/null -w "HTTP %{http_code}" https://httpbin.org/get
HTTP 200
```

**Root cause:** The firewall is a self-imposed jail. The container configures its own iptables rules via `firewall.sh enable`. But `NET_ADMIN` capability, which is required for `firewall.sh` to work, also allows the container to undo those rules. The prisoner has the key.

**Additional finding ‚Äî Root UID bypass:** `firewall.sh` line 167‚Äì170 explicitly exempts UID 0:
```bash
iptables -t nat -A OUTPUT -p tcp -m owner --uid-owner 0 -j RETURN
```
The entrypoint runs as root before `gosu` drops to claude user. Any process that re-acquires root (SUID binary, sudo) bypasses the firewall entirely.

**Additional finding ‚Äî IPv6 bypass:** `firewall.sh` lines 241‚Äì244 use `|| true` on ip6tables rules:
```bash
ip6tables -A OUTPUT -p tcp -m owner --uid-owner "${CONTAINER_UID}" ! -d ::1/128 -j DROP 2>/dev/null || true
```
If ip6tables fails silently (kernel module not loaded), IPv6 egress is unrestricted.

### Recommended Fixes (priority order):

1. **Move firewall to sidecar model** ‚Äî Don't give the agent container NET_ADMIN. Apply iptables rules from a separate privileged sidecar or from the host network namespace via `nsenter`.
2. **Drop NET_ADMIN after setup** ‚Äî Use `capsh --drop=cap_net_admin` before exec'ing the agent process. The capability is only needed during `firewall.sh enable`.
3. **Add `no-new-privileges` securityOpt** ‚Äî Prevents SUID escalation to root UID 0 (which bypasses firewall).
4. **Remove `|| true` from ip6tables** ‚Äî Fail closed on IPv6 rule application failure.

## 4. ATTACK CHAIN ‚Äî OBJECTIVE 2: CONTAINER ESCAPE

### Seam Exploited: Read-write bind mount with no symlink protection

**Attack steps:**

1. Clawker default workspace mode is `bind` ‚Äî mounts project directory into container with R/W access
2. Container reads sensitive files directly: `.env.secret`, `.ssh-fake/id_rsa`, `.git-credentials-fake` ‚Äî all accessible
3. Container creates symlink: `ln -sf /etc/passwd /project-dir/symlink-escape`
4. **Host process follows symlink to HOST /etc/passwd** (not container's)
5. Container creates symlink to host `.ssh/`: `ln -sf /home/josh/.ssh /project-dir/ssh-escape`
6. **Host-side listing shows real SSH keys: `id_ed25519`, `authorized_keys`, `known_hosts`**

**Proof:**
```
$ docker exec container ln -sf /home/josh/.ssh /project-dir/ssh-escape
$ ls -la /project-dir/ssh-escape/   # RUN ON HOST
total 1044
-rw------- 1 josh josh    387 Dec  2 11:43 id_ed25519
-rw-r--r-- 1 josh josh     82 Dec  2 11:43 id_ed25519.pub
-rw------- 1 josh josh    102 Mar 25 16:00 authorized_keys
```

**Root cause:** The bind mount uses `PropagationRPrivate` but this only prevents mount propagation, not symlink following. Docker follows symlinks on the host side when accessing files in the bind-mounted directory. A container-created symlink pointing outside the project directory is followed by the host kernel.

**Impact:**
- Any host process (IDE, backup tool, CI, git) that reads the project directory will follow container-created symlinks
- Container can write arbitrary files to the project directory (code injection, `.bashrc` modification)
- File-level `.clawkerignore` patterns cannot be enforced (only directory-level tmpfs overlays work)

### Recommended Fixes:

1. **Default to snapshot mode** ‚Äî Isolates container from host filesystem entirely
2. **Add `O_NOFOLLOW` / symlink restrictions** ‚Äî Use Linux mount option `nosymfollow` (kernel 5.10+) on bind mounts
3. **Warn on sensitive files** ‚Äî Scan project directory for `.env`, `*.pem`, `.ssh/` before creating container
4. **Document the threat model** ‚Äî Bind mode provides convenience, not isolation. Make this explicit.

**Attack steps (code analysis ‚Äî no live execution needed):**

1. User creates malicious `clawker.yaml`:

4. **Walk-up trust boundary** ‚Äî Build instructions should only come from the project root config, not subdirectory overrides
5. **Schema-level validation** ‚Äî Add `ValidateProjectYAML()` that enforces all constraints at config load time

## 11. ARCHITECTURAL ASSESSMENT

The fundamental tension in Clawker's security model:

**The firewall requires the prisoner to build their own jail.**

`NET_ADMIN` must be granted to the container so `firewall.sh enable` can set up iptables rules. But `NET_ADMIN` also allows the container to tear those rules down. This is architecturally unsound ‚Äî the enforcement mechanism is co-located with the enforcement target.

**The bind mount prioritizes convenience over isolation.**

The default workspace mode (`bind`) gives the container full R/W access to the project directory. This is the correct default for developer experience ‚Äî live file sync is essential. But it creates an inherent tension with the security model that cannot be resolved without moving to a sidecar or volume-based approach.

**Config-to-Dockerfile has no trust boundary.**

The Dockerfile generation pipeline treats `clawker.yaml` as trusted input. When combined with walk-up config discovery (where any subdirectory `.clawker.yaml` can override project config), this creates a supply chain attack vector where a malicious file in a dependency or PR can inject arbitrary build instructions.

**Recommendation:** The most impactful single fix is moving firewall setup out of the agent container entirely ‚Äî either via a sidecar container with `NET_ADMIN`, or by applying iptables rules from the host network namespace via `nsenter`. This eliminates the fundamental "prisoner has the key" problem.
