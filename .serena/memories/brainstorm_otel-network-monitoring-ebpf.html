Problem & Topic

Today's network monitoring pipeline cannot answer "which of my 5 running agents is compromised?" The root cause is architectural: agent attribution belongs at the source-event layer, but logs come from Envoy/CoreDNS which have no agent-identity scope.

User burned hours trying to bolt agent labels onto Envoy access logs / Promtail relabel pipelines. The agent: and project: labels visible in promtail-config.yaml.tmpl are vestigial from those attempts — Envoy never emits those fields, so the labels are always blank.

Real requirement

  • Time-series stream of connection log events, filterable in Grafana's log explorer.
  • Humans filter freely: failures only, single container, single proto, single domain, custom combinations.
  • Per-agent attribution at the source — not joined post-hoc.
  • Cover all L4 protocols — TCP, UDP, SSH, FTP, QUIC, raw sockets.
  • Include bypass-mode traffic (today: forensic black hole).
  • Pure visibility for v1. No baseline learning, no alerts, no “weird domain” classifier. No metric counters — counters give graphs; logs give the filter/transform UX.

Committed Architecture

BPF program (every connect/sendmsg/sock_create decision)
    │  bpf_ringbuf_reserve + submit
    ▼
events_ringbuf  (BPF_MAP_TYPE_RINGBUF, ~1 MB)
    │
    ▼
CP-side reader goroutine  (cilium/ebpf ringbuf.NewReader)
    │  cgroup_id → docker label cache → agent, project
    │  domain_hash → dns_cache reverse map → domain
    ▼
clawker logger.Info()  →  OTel bridge  →  Loki
    │  stream labels: service_name=ebpf-firewall, agent, verdict   (LOW CARD)
    │  metadata:      domain, dst_ip, dst_port, l4_proto, bypass, ts_ns
    ▼
Grafana log explorer  →  humans filter/transform freely

Event struct (BPF C)

struct egress_event {
    __u64 ts_ns;        // bpf_ktime_get_ns()
    __u64 cgroup_id;    // attribution key
    __u32 domain_hash;  // 0 = no DNS resolution
    __u32 dst_ip;       // pre-redirect destination, network byte order
    __u16 dst_port;     // host byte order
    __u8  action;       // 0=allow 1=deny 2=bypass
    __u8  flags;        // bit0=bypass-pass-through, bit1=IPv6
    __u8  l4_proto;     // SOCK_STREAM/DGRAM/RAW
    __u8  _pad[3];
};

Decisions Made

  • Primary signal BPF ring buffer event stream → structured Loki logs. NOT counters. Counters give graphs; logs give Grafana's filter/transform UX. User explicitly chose this shape — visibility/investigation system, not alert/metric system.
  • One log line per L4 decision. bpf_ringbuf_submit at every site that currently calls metric_inc. Userspace reader emits structured log via logger.Info().
  • Loki labels vs metadata. Stream labels = service_name=ebpf-firewall, agent, verdict (low cardinality, stream selector). Structured metadata = domain, dst_ip, dst_port, l4_proto, bypass, ts_ns (high cardinality, | operator filters).
  • Attribution at read time in CP-side reader: cgroup_id → cached Docker labels → agent/project. domain_hash → dns_cache reverse lookup. Why: BPF emits the cheap kernel-scope thing (cgroup_id); CP owns the Docker-label and dns_cache state. Both lookups happen in the same process — no cross-container IPC, no IP-recycling races.
  • metrics_map stays as break-glass debug surface for the existing ebpf-manager dump CLI. Not exported as Prom counters in this change.
  • BPF C changes ARE required. New ringbuf map; new event_emit() helper; ringbuf_submit calls at every decision point. Verifier work minimal (one reserve/submit per site).
  • Drop the "fix Envoy attribution" path entirely. Architectural mismatch; sunk cost.
  • Keep Envoy MITM for enforcement (path_rules) and L7 forensic logs. No longer the primary observability source.
  • Universal L4 coverage via existing connect4/sendmsg4/connect6/sendmsg6/sock_create — TCP/UDP/SSH/FTP/QUIC/raw all covered.
  • DNS attribution = small additive change. dnsbpf plugin (or sibling probe) resolves client_ip → container labels and emits structured DNS query log with agent label. Pairs with ringbuf reader work.
  • Drop instrumentation non-negotiable. Kernel ringbuf drop counter + reader parse failures surfaced via Prom gauge or periodic warn-log. Tetragon convention.
  • Per-cgroup rate limit as v2; v1 trusts ringbuf size + drop counter.
  • Scope = visibility only. No baselining, alerts, classifier. Human eyeballs Grafana log panel.

Conclusions & Insights

  • Architecture The Envoy/Promtail attribution struggle was architectural, not configuration. Identity must be resolved in scope. BPF cgroup_id is in scope inside BPF; CP's reader has the Docker label map in scope. Two layers, both have the info they need without cross-process gymnastics.
  • Bypass forensic black hole fixed for free with eBPF: BPF programs stay attached during bypass and still emit ACTION_BYPASS events. Today bypass = no Envoy logs = blind. Justifies the project on its own.
  • Logs > counters for THIS UX Investigation/filter/transform is what Loki log explorer does. Counters are for graphs and alerts — explicitly not the scope.
  • Loki labels vs metadata agent + verdict as stream labels (low card). Domain / IP / port / proto as structured metadata. Don't repeat the high-card label bug in current Envoy promtail pipeline.
  • L4 vs L7 semantics “Response code” means different things at different layers. L4 outcome (RST/timeout/refused) for everything; HTTP response code only through MITM Envoy. Acceptable since volume isn't the signal — failure-vs-success is.
  • Universal proto coverage Envoy is TCP-only; eBPF at connect4 sees TCP + UDP + raw sockets + dual-stack IPv6. SSH/FTP/QUIC/reverse-shell-on-port-4444 all visible without per-proto config.

Gotchas & Risks

  • BPF verifier complexity new ringbuf map + emit helper means a verifier re-run. Keep event struct flat (no pointers, fixed size). Reserve/submit pattern well-trodden — should pass on first try.
  • Event flood from misbehaving agent tight-loop connect attempts can fill ringbuf. Mitigation order: (1) size ringbuf generously (1 MB initial); (2) surface kernel drop counter; (3) v2 add userspace per-cgroup rate limit with drop emission.
  • Drop instrumentation MUST exist day 1 ringbuf drop counter (kernel) + reader parse-fail counter. Surface via Prom gauge or periodic warn log. Silent drops = security false confidence.
  • CP no-panic invariant applies to reader. Goroutine in CP (PID 1). defer recover(). On unrecoverable error, log event=egress_monitor_unavailable and degrade; do not crash CP. eBPF stays attached.
  • cgroup_id → agent lookup cost Docker inspect per event = too expensive. Cache cgroup_id → {agent, project} with TTL or invalidation on docker-die event. Use existing dockerevents bus.
  • cgroup_id reuse kernel may reuse cgroup_id after container dies. Cache must invalidate on die-event or a new container could inherit old labels.
  • domain_hash reverse lookup dns_cache is keyed by IP, not hash. Reader maintains hash → domain reverse map by periodic scan of dns_cache (16384 max entries; cheap). Direct-IP connects = domain=""; don't drop the event.
  • Loki structured metadata version requires Loki 2.9+. Verify monitor stack pin in internal/monitor/templates/ before depending on structured metadata semantics.
  • Pre-existing: metric_inc silent drop on full metrics_map Uses BPF_NOEXIST; new tuples after 16384-entry max are silently dropped. Worth filing as separate issue (switch to BPF_MAP_TYPE_LRU_PERCPU_HASH, or scraper deletes dead-cgroup entries). Not part of this change but should be noted.

Open Items & Questions

  • Event struct: include raw dst_ip (pre-redirect) so logs show what the agent actually tried? Useful for forensics — yes, locked in.
  • Ring buffer size: 1 MB single global, or per-CPU perf buffer for parallel reads? Start with RINGBUF (simpler, single reader, works on 5.8+); revisit if drops show up.
  • Keep metrics_map + metric_inc alongside event_emit, or replace? Decision: keep for break-glass debug. Cheap.
  • DNS attribution path: extend dnsbpf plugin (CoreDNS side, has client_ip) or new BPF probe at DNS packet level? Probably plugin route; less BPF.
  • Logger OTel bridge structured-metadata wiring: does it emit per-event metadata to Loki, or does it concatenate into the message? Verify before relying on it.
  • Scraper home: internal/controlplane/firewall/ebpf (close to map definitions) or new internal/controlplane/firewall/eventstream/ package?

Next Steps

  1. BPF C changes: add events_ringbuf (BPF_MAP_TYPE_RINGBUF, ~1MB) to bpf/common.h; add event_emit() helper using bpf_ringbuf_reserve/submit; call alongside every metric_inc site in bpf/clawker.c. Define struct egress_event.
  2. Regenerate Go bindings: make ebpf produces updated clawker_*_bpfel.{go,o}.
  3. CP reader goroutine: new file in internal/controlplane/firewall/ebpf (or sibling package). Uses ringbuf.NewReader. Per event: resolve labels via cached Docker lookup; resolve domain via cached dns_cache reverse map; emit via clawker logger with structured fields. Wrap loop with defer recover().
  4. cgroup → agent cache: TTL cache invalidated by dockerevents bus (container die). Lives in CP, near AgentWatcher.
  5. Drop instrumentation: surface kernel ringbuf drop counter + reader parse failures via Prom gauge clawker_egress_events_dropped_total. Warn-log on nonzero.
  6. Wire into cmd/clawker-cp/main.go: subsystem degradation path mirrors wireInitExecutor pattern. On construction error, log event=egress_monitor_unavailable and proceed.
  7. Grafana panels: replace existing Envoy "Egress Traffic" row with log-panels. Default view = all events; pre-built filters for verdict=DROPPED, per-agent drilldown, per-proto, per-domain.
  8. DNS attribution PR (additive, after primary lands): extend dnsbpf plugin to resolve client_ip → container labels and emit structured DNS query log on the same stream.
  9. Separately filed: pre-existing metrics_map silent-drop bug. Not part of this change.
