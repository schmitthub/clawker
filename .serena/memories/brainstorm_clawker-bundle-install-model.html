Problem & Topic

Clawker's extensibility (harness bundles, stacks, monitoring extensions) is currently reference-in-place: yaml registries hold live disk paths, resolution is layered config walkup at use time, and each artifact type grew its own registration/discovery/collision mechanics. The result is convoluted and painful to reason about, even though the underlying concept — community-driven extensibility for any coding agent with sensible shipped defaults — is solid.

GOVERNING MANDATE (user, 2026-07-10): this is a complete overhaul, not an amendment. The premise of this brainstorm is that the prior design was bad. And there was never a "registry/walkup/materialize era" — that design existed only on this unmerged branch as an experiment. It never shipped; no user ever saw it, knew of it, or adapted to it. So there is nothing to migrate, deprecate, stay compatible with, or write upgrade notes for — it is simply code to delete. Nothing from it is grandfathered — no decision, no schema key, no resolution path, no command shape survives on inertia. Implementation MUST NOT take the shortest path through former plumbing: reusing an old registry struct, an old walkup resolver, an old register command's guts, or an old naming/mapping rule because it's already there is a violation, not pragmatism. Every carried-over piece must re-justify itself against THIS model from first principles; default answer is rebuild. When a decision in this doc supersedes an earlier one, the earlier one is dead — do not cite it, do not "maintain compatibility" with it, do not offer it as precedent or justification. Governance: no decisions on unknowns without the user's consent or consultation; every decision is documented here immediately after it is made. Legacy memories were purged 2026-07-10 so no session inherits the old design as bias; what remains says only: legacy experimental code exists on this branch, it proved ineffective, and a full overhaul is planned.

Proposed pivot: adopt an install-by-copy plugin model like agent harnesses use today (Claude Code plugins: one package ships skills + MCPs + commands; install once into an engine-owned store; list = installed). A clawker bundle = 1+ of {harnesses, stacks, monitoring extensions} — peers, no hierarchy. Sources = local path or git. There is no marketplace concept in this model. This brainstorm works out the model: shipped-defaults handling, store layout, naming collisions, versioning/updates, dev loop, and how far this reverses the shipped stack-contract (docs/stack-contract.mdx).

The Design (current state)

A bundle is how third parties ship 1 or more of {harnesses, stacks, monitoring extensions}. The three component types are peers inside a bundle — a stack never lives inside a harness; a bundle needs no harness at all. Bundles exist ONLY for distribution. Everything else resolves from two non-bundle tiers. Three tiers total:

TierWhatNamesLifecycle
Embedded floorShipped harnesses (claude, codex), stacks (node, go, …), monitoring extensions (claude-code) baked into the binary as standalone things — no bundle wrapper, no disk materializationBare, reserved, unclaimableUpdates atomically with binary upgrade
Loose localHand-rolled component dirs dropped into convention dirs; zero ceremony — no install, no registry write, no wrapper manifestBareExists while the dir exists
Installed bundlesDistributed content declared in bundles:, fetched to host cacheQualified namespace.bundle.componentExplicit install/update/remove

Bundle directory layout (dir convention, no path fields)

my-bundle/
├── .clawker-bundle/
│   └── bundle.yaml            # bundle manifest — pure metadata, nothing else
├── harnesses/
│   └── <name>/                # dir name IS the component name
│       └── harness.yaml       # component manifest (+ whatever the component ships)
├── stacks/
│   └── <name>/
│       └── stack.yaml
└── monitoring/
    └── <name>/
        └── monitoring.yaml
  • Components discovered purely by convention dir; unknown top-level dirs → advisory warning, not error.
  • Any subset of the three dirs; 1+ components total. A single-component bundle is just a repo with one convention dir (stacks/mystack/) + the marker dir — NO bare-manifest-at-root special case (user ruling 2026-07-10: bundle files follow the dir convention exactly like the loose tier; the earlier "implicit single-thing bundle" idea is dead). One enumeration path; all three tiers structurally identical.
  • harness.yaml is a component manifest only — it no longer doubles as a bundle manifest.

Bundle manifest schema (.clawker-bundle/bundle.yaml)

namespace: acme         # REQUIRED (distributed). Maintainer branding — org/handle/umbrella, whatever they prefer. charset a-z0-9-. NOT a marketplace — pure manifest metadata, no listing concept.
name: acme-tools        # REQUIRED. charset a-z0-9- (no _). Never derived from source coordinate.
version: 1.2.0          # optional; absent → resolved source sha IS the version
description: …          # optional
author: …               # optional
repository: …           # optional
license: …              # optional

No component path fields — enumeration is the dir convention above. Bundle identity = (namespace, name) pair + convention dir names for components, exclusively; owner/repo/URL segments are never identity-bearing. Namespace is self-declared (no central registry enforces scope ownership — npm-scope-shaped without the registry); two strangers claiming the same (namespace, name) = hard error at fetch/resolve, accepted as rare residual.

Loose-tier convention dirs

project:  .clawker/{harnesses,stacks,monitoring}/<name>/
user:     <config-dir>/{harnesses,stacks,monitoring}/<name>/

Component's own manifest is the only requirement (the SKILL.md analog). Drop dir → exists, bare-named.

Declaration & content flow

bundles:                                      # clawker.yaml key, ANY layer (user base / project / .local), union-merged
  - url: https://git.example.com/acme/tools.git
    ref: v1.2.0                               # ref or sha REQUIRED on remote sources
  - url: git@github.com:acme/mono.git
    path: bundles/tools                       # subdir of a repo (monorepo case)
    sha: 4f2a…40-char                         # sha beats ref when both set
  - path: ./vendor/my-bundle                  # local: loads IN PLACE, no cache copy = the dev loop
  • Sources are git-generic: any host, https or ssh honored as provided. Provider shorthands (owner/repo) = optional sugar expanding to a clone URL, never canonical, never identity.
  • Remote content fetches into host-global cache <data>/bundles/<name>/<version>/ — versions coexist (project A pins v1, project B v2). Project root never holds fetched content.
  • Fetch = explicit user intent only. Declared-but-missing → prompt/error naming the command; never silent. clawker bundle install <src> = sugar: writes the yaml entry + pre-fetches.
  • Ref drift accepted (CC model): no lockfile, no sha write-back; branch/tag entries move only at explicit bundle update; reproducibility = pin sha.
  • Auto-update: opt-in per entry, OFF by default; check fires at start of bundle-consuming commands (build/run/monitor up), CC compare-then-refetch on version change only.

Naming & addressing

  • Name charset a-z0-9- (no underscore) for bundle and component names.
  • Bare name (node) = embedded floor or loose local. Qualified = installed-bundle content, ALWAYS 3 dot-separated segments: namespace.bundle.component (acme.tools.node). Globally unique by construction — no declared-set uniqueness rules, no addressing collisions possible. Dot outside name charset → bijective splits.
  • ONE spelling everywhere: yaml selection keys, CLI args, @: selector, image tags, volume names, index names. No alternate colon form, no mapping layers.
  • Bundle identity = (namespace, name); same pair from two sources = hard error at fetch/resolve (C1 resolved). C2–C4 resolved; C5 open.

Selection (what survives in config)

  • Config holds SELECTION only, by possibly-qualified name — zero paths, zero registries: build.stacks: [node, acme.tools.node], per-project harness pick, monitoring selection in clawker.yaml layers.
  • Enable/disable = selection itself (CC enabledPlugins / codex enabled analog). No enabled: flag; bundles never activate as a unit. Deselect = disable (cache persists); delete bundles: entry = unavailable; bundle remove = purge.
  • Monitoring consumption = option D: monitor up idempotently seeds the host stack from the cwd-resolved projection (layers + loose dirs + floor); collector regen uses the all-ever-seeded ledger union; removal = volume prune.

Commands

clawker bundle install | list | remove | update | validate [--strict] — FINAL verb set; replaces all register-era commands. No eject verb — customizing embedded content = copy source from public repo into a loose dir (C3 shadow). Validation split: hard-fail = unreachable source, malformed/missing bundle.yaml, missing required fields, type mismatch; warn = unknown dirs/fields (typo suggestions), empty component dirs.

Decisions Made

  • Shipped = embedded Shipped harnesses/stacks/monitoring extensions stay embedded in the binary — no materialization to disk, no fetch-from-repo. They are the virtual pre-installed floor: name-addressable, listed with source built-in, updated atomically with binary upgrades. Why: with install-by-copy there is no strict pathing contract left to satisfy — nothing needs shipped content to exist at a disk path. Embed keeps out-of-box offline guarantee, kills the entire materialize/overwrite/self-heal debate (binary upgrade IS the update), avoids repo-fetch pinning/network/version-skew machinery, and matches existing stack-contract law ("shipped definitions resolve straight from the embedded FS"). How to apply: front doors accept names for shipped things (not just paths); resolution = install store > embed floor.
  • Addressing = qualification (codex / Claude Code route) Components inside an installed bundle are ADDRESSED as qualified names (spelling = namespace.bundle.component per the C1a + separator locks below); bare names reserved for the shipped/embedded floor and loose local tier. pi-style precedence algebra rejected — precedence-resolution across installed content is the same walkup implicitness this pivot kills. Why: grounded in real prior art (codex plugins cache {marketplace}/{name}/{version} + qualified skill names; Claude Code plugin:component prefixes). Explicit beats terse for debuggability. How to apply: this is an ADDRESSING decision only. It does NOT decide collision handling — no collision-handling decision has been made anywhere in this brainstorm (user, 2026-07-10); all collision questions live in Open Items. An earlier Docker-convention owner/name identity scheme was superseded by this entry and is dead.
  • Terminology: BUNDLE Clawker's unit is a bundle, not a plugin — bundles add material (build inputs, observability assets), they do not extend clawker CLI execution. A bundle contains any mix of {harness, stacks, monitoring extensions}. The existing harness-bundle concept EXPANDS to this — a bundle no longer requires a harness. Why: honest naming — no code runs in the engine (engine-sovereignty law preserved); "plugin" would overpromise execution hooks.
  • Local convention-dir tier Alongside installed bundles, loose local resources auto-load from convention dirs at project level and user level (codex skills model: .codex/skills project / ~/.agents/skills user). No install step, no registry entry — drop a stack/extension dir in the right place and it exists. This REPLACES register-by-path entirely. Why: personal/local iteration needs zero ceremony; packaging is only for distribution. Exactly how codex/Claude Code treat local skills vs marketplace plugins.
  • Shipped = standalone baked-in things, NOT a bundle Baked-in harnesses (claude, codex), stacks (node, go, …), and monitoring extensions (claude-code) are individual bare-named things on the embedded floor. There is no "shipped bundle" wrapper. Bundle exists ONLY as an install/distribution unit (collection of 1+ things; any thing also installable standalone). Why: user ruling 2026-07-10 ("we don't need to ship a bundle… they are baked in"). Kills the shipped-component-qualification question — shipped things are bare, qualification applies only to installed-bundle content.
  • Manifest = pure metadata; components by dir convention .clawker-bundle/bundle.yaml holds name/version/description/author/repository/license ONLY — no component path fields. Components discovered from convention dirs: harnesses/<name>/, stacks/<name>/, monitoring/<name>/; dir name IS the component name. Unknown top-level dirs → advisory warning, not error. No bare-manifest-at-root mode (superseded 2026-07-10: single-component bundle = one convention dir + marker dir, same enumeration path as everything else). harness.yaml becomes a component manifest only — stops doubling as bundle manifest. Why: user ruling 2026-07-10 ("yea we just do the same. dir convention") after grounding: CC and codex are both convention-first with optional path overrides most plugins never use; clawker goes one simpler at v0 (overrides addable later, purely additive). The old "explicit, never convention" ruling governs runtime copy behavior (seeds/staging), not content enumeration.
  • Inclusion flow: declare in clawker.yaml, content in host cache bundles: is a clawker.yaml key at ANY layer (user-level config-dir clawker.yaml for personal cross-project bundles, project clawker.yaml for team bundles, .local for private), union-merged across layers — settings.yaml is NOT involved (it is app/host settings, not a config layer). Entries are source specs: {github: owner/repo, ref|sha}, git url, git-subdir, {path: ./dir}. Remote sources fetch into the host-global cache <data>/bundles/<name>/<version>/ (versions coexist; project A pins v1, project B v2). Path sources load IN PLACE, no cache copy — that IS the dev loop (no --link needed). Project root never holds fetched content; clawker.yaml is committable and teammates converge automatically. clawker bundle install <src> = sugar: writes the yaml entry + pre-fetches. Why: exact CC model (project settings reference name@marketplace, content lands in user cache, trust prompt on first use); opencode/pi identical in spirit. Declaration/content split keeps repos clean and sharing trivial.
  • Monitoring = option D: idempotent seed-at-up from cwd (LOCKED 2026-07-10) No host enable registry. monitor up is idempotent: stack down → start + seed; stack running → seed missing extensions onto it. Active set = projection of resolved project config at up-time (cwd layers + convention dirs + embed floor). Seeded artifacts persist until volumes pruned; next up re-seeds from cwd. settings.yaml exits the monitoring story; selection lives in clawker.yaml layers. Scope-tension candidates A/B/C all obsolete. Why: everything an extension ships except collector routing is an idempotent REST apply against the running stack (templates/pipelines/ISM/saved-objects; verified against internal/bundler/monitoring_unit.go payload); collector routing = config regen + one-container recreate, cheap. So consumption can be an at-up-time projection — no pinned host registry, no dangling refs, tension dissolves. How to apply: collector regen MUST use union of all-ever-seeded units (ledger: UnitsMarkerFile, internal/monitor/units.go:20), not just cwd's — else project B's up drops project A's routings. Removal = volume prune only.
  • Auto-update: offered, OFF by default, CC-mirrored (LOCKED 2026-07-10) Opt-in auto-update exists but defaults off. Update determination and timing MIRROR CC — refetch only on a detected version change (declared version bump, or new commit when version is the source sha), never a greedy check on every clawker invocation. Only meaningful for ref/branch entries; sha-pinned never moves. Why: user lock 2026-07-10 — offer the convenience, keep host writes intentional by default, and don't turn every CLI call into a network probe. How to apply (CC facts, grounded 2026-07-10): CC checks at SESSION STARTUP only — "Background auto-updates run at startup", never per command; flow = refresh listing → resolve version → compare → refetch only on difference ("if the resolved version matches what a user already has, update and auto-update skip the plugin"). Version resolution precedence: manifest version field → declaring-entry version → git commit sha of source. Clawker mapping: version = bundle.yaml version → else resolved source sha (same precedence, no marketplace layer). Clawker has no long-lived session, so the "startup" analog is OPEN: candidate = TTL-gated check only at bundle-consuming front doors (build/run/monitor up), never mid-operation, plus explicit bundle update always available.
  • Fetch = explicit user intent only (LOCKED 2026-07-10) Nothing downloads to the host without the user saying so. Declared-but-missing remote bundle → prompt/error pointing at the command, never silent fetch. clawker bundle * management commands added; existing register-era commands refactored into them.
  • ref/sha OPTIONAL on remote sources — unpinned tracks the repository's default branch (RATIFIED 2026-07-12, overrules the 2026-07-10 ref/sha-REQUIRED clause) A url-only source is legal: install clones the remote HEAD (default branch tip); bundle update — and auto_update: true at bundle-consuming commands — resolves the remote HEAD and refetches on movement, exactly like a branch ref. sha still beats ref; sha-pinned never moves; explicit empty ref: "" stays rejected. Fetch remains explicit-intent (unpinned ≠ silent fetch). Why: user ruling 2026-07-12 ("ref isn't supposed to be required... do whatever CC/codex does since it'll be what avg user expectations are"); CC-literal, doc-verified: marketplace/plugin source ref is "Optional. Git branch or tag (defaults to repository default branch)", "every new commit is treated as a new version". A latest-tag/semver reading of "most recent release" was considered and dropped — CC uses semver only for dependency constraints.
  • Distribution = bundle only; local = loose tier, zero ceremony (LOCKED 2026-07-10) Anything DISTRIBUTED travels as a bundle (install-by-copy, qualified names). Anything LOCAL is loose-tier: hand-roll/download a component dir into a convention dir and it exists immediately — no install, no registry write, no wrapper manifest; component's own yaml is the whole contract. Enable = the existing selection surfaces only (build.stacks, harness pick, monitor up projection). Loose things are bare-named like shipped things; installed-bundle content is qualified — collision-free by construction between loose and bundled. Why: exact two-tier shape of both CC and codex. CC: loose skills (.claude/skills/foo/SKILL.md, no manifest, in-place, bare /foo) vs plugins (always plugin:-namespaced, cached). codex: .codex/skills/~/.agents/skills auto-load vs plugin:skill cache installs. Neither ever collides loose-vs-plugin because namespacing differs. How to apply: dev loop for a distributable bundle = path: source (in-place, CC --plugin-dir analog but persisted via yaml); freeze/pin a local dir = bundle install ./dir (copy, codex-style). Both coexist.
  • Convention-dir layout (LOCKED 2026-07-10) Project .clawker/{harnesses,stacks,monitoring}/<name>/; user <config-dir>/{harnesses,stacks,monitoring}/<name>/. Component's own manifest (stack.yaml/harness.yaml/monitoring.yaml) is the only requirement — the SKILL.md analog. Drop dir → exists, bare-named.
  • Testing mandate: real integration tests with fakes (LOCKED 2026-07-10) Implementation ships with PROPER integration tests: an in-process git mock/fixture server serving real fetches over BOTH http and ssh; real-filesystem temp environments (no in-memory fs theater); tests construct and invoke the REAL production code paths end-to-end (declare → fetch → validate → cache → resolve → select); seams only where genuinely necessary, per existing seam law (nil in prod, never drive prod). Explicitly banned: tautological/self-serving assertions, hand-rolled fixtures shaped to make the test pass, mocking the very layer under test. Why: user ruling 2026-07-10. Fetch/install/resolve is the core of this model — golden files and unit-level manifest parsing prove nothing about the journey. Aligns with standing rules [[feedback_integration_tests_drive_real_journey]], [[feedback_no_happy_path_fixture_theater]], [[feedback_no_test_seams_in_signatures]]. How to apply: git server fake must cover http and ssh transports (ref/sha resolution, subdir checkout, unreachable-source and malformed-manifest failure paths — the hard-fail matrix above), live under the standard test infra layout, and be reusable across bundle/install/update/validate suites.
  • Identity = manifest + convention ONLY; sources are git-generic (LOCKED 2026-07-10) Bundle identity comes exclusively from the self-declared bundle.yaml name (validated against clawker charset at the front door) and component identity from convention-dir names. NOTHING is ever derived from the source coordinate — no owner, no repo name, no URL segment. Clawker supports git, not specific providers: GitHub/GitLab/Gitea/private hosts all reduce to a clone URL + optional subdir; their varying naming schemes and pathing nuances are never normalized or chased. Validation of manifests/conventions mirrors CC's model (hard-fail structural errors at front door, advisory warnings for unknowns, bundle validate --strict for authors). Why: user ruling 2026-07-10 — many VCS/SCM providers exist, public and private; provider-charset leakage (GH repos allow _ ., GitLab uppercase) would poison the name model. CC precedent: plugin name = manifest field, repo irrelevant.
  • Bundle sources: git-generic, CC-shaped (LOCKED 2026-07-10) Canonical source = a git clone URL, any host, any protocol: {url: https://…|git@…, ref?, sha?} — https and ssh honored exactly as provided; {url, path: sub/dir, ref?, sha?} for a subdir of a repo (monorepo case); {path: ./dir} for local in-place. sha = full 40-char commit and beats ref (branch/tag) when both set. Provider shorthands (e.g. owner/repo) are OPTIONAL sugar that expands to a clone URL — never the canonical form, never identity-bearing, and docs/examples lead with the generic URL form. Why: user rulings 2026-07-10 — clawker supports git, not providers; CC's source-spec shapes (plugin-marketplaces.md "Plugin sources") adopted for field semantics (ref/sha precedence, subdir, local path), minus provider-centrism. How to apply: validation mirrors CC split at the write front-door: hard-fail = unreachable source, malformed/missing bundle.yaml, missing required fields, type mismatches; warn = unknown dirs/fields (typo suggestions), empty component dirs; clawker bundle validate ./dir [--strict] for authors.
  • Ref drift accepted, no lockfile (LOCKED 2026-07-10) No resolved-sha recording anywhere — not in yaml, not in a lockfile. CC model verbatim: cache is version-keyed; a branch/tag ref drifts only at explicit bundle update; teammates on a ref entry may hold different commits until their next update; anyone needing reproducibility pins sha in the entry (already first-class). Why: user lock 2026-07-10 — accept drift like CC; no second committed artifact, no yaml rewriting behind the user's back.
  • Meta-rule: tension → defer to CC/codex (LOCKED 2026-07-10) When a design tension arises in this model, the resolution is CC's or codex's approach, literally — do not invent a novel mechanism. Applied immediately to the auto-update trigger: the OPEN "startup analog" resolves to the literal mapping of CC session-startup / codex startup-sync — the check fires at start of bundle-consuming commands (build/run/monitor up), opt-in entries only, CC's compare-then-refetch flow (version change only), never mid-operation, no invented TTL layer. bundle update always available. Why: user ruling 2026-07-10 — "literally do what CC or codex does… that's basically your only options when hitting tension." They solved these problems already; novel mechanisms were rebuked earlier in this brainstorm too.
  • C1 resolved: two-dimension identity + hard fail (LOCKED 2026-07-10) bundle.yaml gains a REQUIRED namespace field: maintainer-entered branding (org, handle, umbrella — "whatever branding they prefer"), same a-z0-9- charset. It is NOT a marketplace — no listing/distribution concept attaches to it; pure manifest metadata, self-declared by the same maintainer in the same file. Bundle identity = the (namespace, name) pair. Two different sources resolving to the same (namespace, name) = hard error at fetch/resolve time (declare-time impossible — yaml layers union-merge lazily), error names both sources + remedy; no user alias; same-source re-declare idempotent. Why: user ruling 2026-07-10 — earlier hard-fail rejection targeted single-dimension declared names, where "codex"/"go"-style names guarantee collisions; a second self-branded dimension is the due diligence that makes hard-fail residual-rare (npm scopes / CC's arbitrary-branding marketplace names, minus the registry). Grounding: CC has no defined non-reserved duplicate behavior (open bug #45266, broken qualifier #20593 — uncopyable); codex hard-errors duplicate marketplace names at add with no alias (validate_marketplace_name_for_add) — the one defined prior-art answer, mirrored here at clawker's detection point. How to apply: collision check compares source coordinates of all declared bundles whose manifests resolve to the same (namespace, name); differing sources → error listing both yaml entries (layer + source) and telling the user to drop one. Sub-questions C1a–C1c resolved in the entries below.
  • C1a resolved: addressing = namespace+bundle+component, 3 segments (LOCKED 2026-07-10, supersedes same-day 2-segment ruling) Qualified address carries ALL THREE: namespace, bundle name, component name (e.g. acme.tools.codex modulo separator, below). Globally unique by construction: (namespace, bundle) is identity-unique (C1) and component names are unique within a bundle per type dir. This DELETES the interim "declared bundle names unique per resolved set" hard error — acme/codex and zorg/codex coexist declared and addressable. Namespace additionally keys identity surfaces: cache <data>/bundles/<namespace>/<name>/<version>/, collision detection, errors, update matching. Why: user ruling 2026-07-10. Component-to-namespace is NOT 1:1 — one maintainer's bundles routinely ship same-named components (acme/codex-tools + acme/codex-extras both shipping harness codex is the normal case), so dropping bundle from the address (2-segment namespace__component) is guaranteed to fail; and bundle-name-only addressing (CC/codex-literal plugin:skill) is those systems' documented open hole (#45266 — bundle names cluster on obvious words like "codex", collision constant not rare; CC's own proposed fix is scoped addressing). Meta-rule inapplicable: prior art broken here, its own issue tracker points at this shape. How to apply: bare name = floor/loose; presence of separator = installed-bundle content, always exactly 3 segments; malformed segment counts = front-door validation error.
  • C1b + C1c resolved: namespace required; clawker reserved (LOCKED 2026-07-10) namespace REQUIRED in every bundle.yaml (CC and codex both require marketplace name; every distributed bundle has .clawker-bundle/bundle.yaml — no manifest-less bundle form exists). Reserved namespaces: clawker + obvious impersonation combos rejected at validate front door (CC-literal, #18329; codex reserves nothing — CC's is the one defined answer).
  • Enable/disable = selection keys, no flag (LOCKED 2026-07-10) CC enabledPlugins / codex [plugins."x@y"] enabled ARE those systems' selection surfaces; clawker's existing selection keys (build.stacks, harness pick, monitoring selection in clawker.yaml layers) are the literal analog — components activate by being named, full stop. No enabled: flag on bundles: entries; bundles never enable/disable as a unit (distribution truck only). Deselect = disable (cache persists, CC disable-keeps-cache analog); delete bundles: entry = unavailable; bundle remove = purge cache.
  • C2 resolved: cross-type same names coexist, zero validation (LOCKED 2026-07-10, meta-rule) CC and codex both perform NO cross-type duplicate check; types are separated by their consumption surfaces. Clawker mirror: a bundle may ship stacks/node/ AND harnesses/node/; loose/floor tiers likewise. Legal because every clawker surface is type-scoped by construction: build.stacks accepts only stacks, harness pick only harnesses, monitor up projection only monitoring; docker artifacts inherit type context from the surface that produced them. bundle validate checks nothing cross-type. Why: meta-rule — both prior arts tolerate by construction; inventing a uniqueness rule neither has = novel mechanism, banned. How to apply: standing invariant — any FUTURE surface that merges component types in one namespace (a combined list, a shared index) MUST carry type context in its naming; that is the property both prior arts rely on.
  • C3 resolved: loose local shadows embedded floor (LOCKED 2026-07-10, meta-rule) Same bare name in a loose convention dir and on the embedded floor → loose wins. CC-literal: "plain skills override bundled skills".   • Update-failure semantics: cache keeps serving, never purged on failed fetch (LOCKED 2026-07-10, meta-rule) Source unreachable/moved/deleted at bundle update (or the opt-in front-door auto-check) → error/warning surfaced naming the source and reason; the cached installed version keeps serving untouched. Why: meta-rule, grounded 2026-07-10. codex: failed git sync → cached version prioritized "to maintain stability", warning logged, no purge on failure. CC: cache/enable orthogonal, version-keyed cache; unreachable marketplace = surfaced troubleshooting error while installed plugins keep working.

There is NO bundle-internal sibling-stacks/-first resolution lane — that was the old bundle-embedded-stacks layer and an interim same-day ruling resurrecting it was struck as legacy. Why: user rulings 2026-07-10 — deps must be able to reference bundled stacks (qualified addressing already covers it) and sibling-first was rejected as legacy resolution ("incapable of abandoning legacy designs" rebuke).   • Separator = . dot (LOCKED 2026-07-10, supersedes same-day __ lock) Qualified name = namespace.bundle.component — acme.tools.codex — ONE spelling on every surface: yaml selection keys, CLI args, @:acme.tools.codex selector, image tags (clawker-proj:acme.tools.codex), volume names, index names. No colon form, no translation/mapping layers. Bijective: name charset (a-z0-9-) contains no dot, so splits are unambiguous; always exactly 3 segments when qualified. Why: user pick 2026-07-10 ("dots are better") — __ twice per name rejected; dot matches clawker's existing clawker.project.agent hierarchy grammar. Legality: OCI tag regex [a-zA-Z0-9_][a-zA-Z0-9._-]{0,127} permits dots (non-leading; our names start a-z0-9); LIVE-VERIFIED 2026-07-10 — docker volume create clawker-septest-acme.tools.codex-config and docker tag … clawker-septest:acme.tools.codex both accepted; OpenSearch index names allow mid-name dots (logstash-date convention).
  • Relative path source = relative to the DECLARING FILE's directory, every layer (RATIFIED 2026-07-12, supersedes the provisional config-dir-absolute rule) One uniform rule: an absolute path: is used verbatim; a relative one resolves against the directory of the clawker.yaml layer that declared it — project layer, local overlay, and user config-dir alike. No layer special case, no validation ban, isConfigDirLayer deleted. The CLI (bundle install ./dir) re-anchors its cwd-relative argument at write time: filepath.Abs from cwd, then filepath.Rel against the target file's directory (absolute fallback when no relative form exists), so committed project files stay portable and a user-layer entry round-trips regardless of invocation cwd. Why: user ruled the provisional layer special case "adding complexity" — and the uniform rule is the meta-rule answer verbatim: codex marketplace.json resolves relative sources "against the root directory of the marketplace.json file itself", and codex's CLI canonicalizes relative→absolute at config-write time (deepwiki-grounded 2026-07-12). Project clawker.yaml sits at the project root, so the old relative-to-project-root behavior is unchanged where it mattered. How: resolver resolveLocalPath(src, decl.File) anchors at filepath.Dir(File); rewriteLocalPath in cmd/bundle/install does the CLI re-anchor (with ~/$VAR expansion).
  • Bundle source sha = full 40-char lowercase hex, rejected otherwise (PROVISIONAL 2026-07-10, implementation fill — flagged for user ratification) Abbreviated or uppercase SHAs are load errors, not normalized. Why: rejecting at the front door means the fetch engine and cache paths never canonicalize; git emits lowercase, so the rule costs users nothing. Spec locked only "full 40-character sha"; the reject-don't-normalize stance is an implementation fill.
  • C1 is uniform: in-place declaration vs installed cache entry = hard error, never a silent winner (RATIFIED 2026-07-12, overrules the provisional in-place-wins fill) Two sources claiming the same (namespace, name) — whether two declarations, or a declared path: source and an installed cache entry — are a CollisionError naming both source coordinates, both locations, and the remedies: remove one, clawker bundle remove <ns>.<name> to purge the cached copy, or change the namespace. Author dev flow is explicit: purge the cached copy of your own bundle before developing it via path:. Why: user overruled the silent override — identity is self-declared, so the resolver has no evidence two same-identity sources are the same bundle; a local dir silently standing in for an installed identity is an attack primitive (attacker mirrors an expected bundle's namespace/name to hijack its resolution, including a harness's egress floor). "You simply can't have two bundles installed, path or git based, that share the same namespace and names — that was the whole point of adding them." (user, 2026-07-12) How: resolver mergeCachedBundles errors on a cache identity already claimed in-place (cache side's coordinate read best-effort from source.yaml, cache:<dir> fallback); install-time cache-side C1 vs source.yaml already existed.
  • Build-path provenance = winning source only; shadow listings live in bundle list (PROVISIONAL 2026-07-10, P4 implementation fill) A single Resolve reports the winning tier without computing shadows — computing them would scan the installed-bundle set, violating the "broken bundle declaration must not block a floor-only build" invariant. Build output prints one line per non-floor resolution (stack node ← project (…), stack acme.tools.node ← bundle acme.tools); the full shadows … vocabulary is List/bundle list output.
  • Selection keys accept qualified spellings at the config front door (2026-07-10, P4 gap fix) build.stacks entries, build.harnesses: overlay keys, and harnesses: init-config keys validate via consts.ValidateComponentRef/ValidateHarnessRef (bare rule + reserved image-tag aliases bare-only, or structural 3-segment). The P4 build left the old bare-only rules in config while bundler resolved qualified — bundled components were unselectable end to end; caught in orchestrator review.
  • Bundle identity spelling = dotted namespace.name everywhere user-facing (2026-07-10, P3) BundleID.String(), collision errors, listings, and the bundle remove/update <namespace.name> arguments all use the dotted 2-segment form — the natural prefix of the qualified component address. Filesystem cache layout joins the pair as path segments separately; a slash spelling never surfaces. Why: one-spelling rule extended to identity — the P3 build initially rendered ns/name, which would have put a second spelling next to the planned dotted CLI args.
  • monitor init renders the cwd projection only — never reads or writes the persisted units ledger (PROVISIONAL 2026-07-10, P5 implementation fill) init merges the current projection into an empty in-memory ledger and renders its union; monitor up alone performs the all-ever-seeded merge and persists. Why: init is an optional pre-render scaffold that can run before any up; a scaffold mutating host ledger state would be surprising. Spec was silent on init↔ledger.
  • Collector recreate fires only when otel-config bytes CHANGED against a prior render — a first-ever render never recreates (2026-07-10, P5 implementation fill) Implemented as stop+remove of the collector before the single compose up (compose never recreates on bind-mount content change), so the flow is identical in detached and foreground modes. The seeded projection persists via a flock-guarded load-merge-save (SeedLedger) after a successful up, so concurrent ups from two projects cannot lost-update each other's ledger seeds.
  • `bundle list` renders honest per-identity rows linking declarations to the cache (RATIFIED 2026-07-12, supersedes the P6 count-hint fill) Manager.Statuses() joins the declaration side (BundleDeclarations canonicals) to the cache side (source.yaml canonicals): one row per identity/declared source — resolving (installed/in-place, with declaring file + selected version), declared-but-not-installed (identity unknown until fetched, shown by source coordinate), cached-but-not-declared (re-declare or bundle remove), and hand-placed/unmanaged (never resolves; remove). Rendered as a second table in the default output; the actionable states repeat as stderr hints in every output mode. The count hint is retired. Why: source.yaml now exists, so the honest per-identity reconciliation the count-hint deferred is buildable — and declaration-gating makes the undeclared/unmanaged states user-visible facts that need surfacing.
  • owner/repo install shorthand expands against github.com, normalizing a trailing .git (2026-07-10, P6 implementation fill) CLI-only sugar: acme/tools and acme/tools.git both expand to https://github.com/acme/tools.git before the yaml write; a repo name merely containing a dot keeps its dot. The persisted bundles: schema stays git-generic — no provider field, no shorthand ever stored.
  • Cached bundles are DECLARATION-GATED at resolve (RATIFIED 2026-07-12, overrules the P8 not-gated fill) A cached bundle resolves ONLY while a live remote declaration's Canonical() matches its source.yaml; deleting the bundles: entry makes the cached copy inert (kept on disk until bundle remove; re-declaring reactivates instantly, no refetch). A cache entry with no source.yaml (hand-placed) never resolves — no ghost sources: everything resolvable traces to explicit config. Version selection follows the matched source's latest fetched_at, replacing last-sorted. The in-place-vs-cache C1 correspondingly narrows to BOTH-declared — an undeclared cache entry cannot collide, so the author dev flow is swap url→path declaration, no purge. Why: the un-gated fill contradicted the locked enable/disable clause "delete the bundles: entry = unavailable", and a resolvable-but-undeclared cache is a ghost source the user cannot see in config — the user overruled it once source.yaml made the linkage buildable ("why would an undeclared bundle still resolve?").
  • Update reports per-bundle typed outcomes (2026-07-10, P8 implementation fill) Manager.Update(ctx,id) ([]UpdateResult, error) with UpdateOutcome ∈ {sha-pinned skip, unmanaged skip (hand-placed cache entry with no source metadata / no movable ref), unchanged, refetched, failed-cache-kept}; the top-level error is cache-enumeration only. AutoUpdateCheck maps the same outcomes to warnings (refetched/failed warn, skips silent).
  • Fetch-pipeline hardening (2026-07-10, P8 review) A declared monorepo subdir is guarded against BOTH spelled traversal (filepath.IsLocal) and symlink indirection (EvalSymlinks containment) — a hostile repo cannot point the load-and-copy pipeline outside its clone. A manifest version is rejected when dot-prefixed (invisible to the cache scan) or equal to source.yaml (would clobber cache metadata). Conclusions & Insights

  • Root cause The convolution fault line is reference vs install, not the number of artifact types. Reference-in-place forces dynamic conditional discovery (cwd-dependent visibility), N collision front doors, portability caveats, and the materialization fight. Install-by-copy makes list = installed, period.
  • Update problem dissolves The "must always overwrite or we can't update" problem was an artifact of reference/materialize semantics. Embedded shipped content updates with the binary; installed copies update by explicit re-install from recorded source; user edits belong in the source (or a fork), never in the store.
  • Prior-art convergence (codex / opencode / pi, researched 2026-07-10) All three open harnesses: install-by-copy into tool-owned store keyed by SOURCE identity (marketplace/npm/git); manifest declares contents by relative path with convention-dir fallback; multi-resource packages are normal; bare spec tracks latest, pinned stays pinned; local convention dirs coexist with installed packages. None uses live path-reference registries resolved by config walkup. codex: plugins/cache/{marketplace}/{name}/{version}, plugin:skill qualification, name@marketplace config refs. opencode: npm packages, --ignore-scripts (no install-time code exec — matches engine sovereignty). pi: npm/git/local sources, pi manifest key declaring resource paths, precedence tiers (rejected for clawker).
  • Registry era ends Local convention dirs + installed bundles + embedded floor replace ALL yaml registries. Project/settings yaml keeps only SELECTION (which stacks/harness/extensions to use, by possibly-qualified name) — zero paths in config.
  • Config-surface law (user, 2026-07-10) clawker.yaml = the layered cross-project config surface (user-level config-dir file is the BASE layer, project overlays, .local on top — CC settings-scope model). settings.yaml = pinned host storage struct for things that must NOT be influenced per project. Consequence: bundles: (availability) lives in clawker.yaml layers; build/harness selection lives in project clawker.yaml; monitoring selection ALSO lives in clawker.yaml layers per the later option-D lock (settings.yaml exits the monitoring story entirely). Every surface justified by scope.
  • Install/enable semantics researched (CC docs + codex source, 2026-07-10) CC: fetch ONLY at explicit plugin install/update; cache keyed by version — explicit semver refetches only on version bump (new commits without bump = no update), omitted version → git SHA becomes the version → every commit is a new version; disable keeps cache (writes false), uninstall deletes cache + settings entry; auto-UPDATE is real and distinct from auto-install: per-marketplace toggle, DEFAULT ON for official Anthropic marketplaces, autoUpdate: true managed setting — refetches already-installed plugins on version change; auto-INSTALL of never-installed plugins never happens. codex: explicit install PLUS startup-sync reconcile (downloads missing/version-changed bundles for configured plugins); enabled=false in config.toml keeps cache. Both: enable state is config, content is cache — orthogonal.
  • Local-fs resolution researched (CC docs + codex, 2026-07-10) CC loose tier: SKILL.md in convention dir = whole contract, no manifest, in-place, bare name; scope precedence for same-named loose skills = enterprise > personal > project, and within a tier "plain skills override bundled skills" — LOCAL-WINS precedent for our local-vs-shipped question. CC local plugin install: --plugin-dir = in-place dev loop (no cache, session-scoped); marketplace relative-path source = copy-to-cache. codex: no in-place flag at all — even plugin install ./dir copies to cache. Skills-dir plugin (.claude-plugin/plugin.json inside a loose skill dir) upgrades a loose dir to bundle more component types while staying in-place — precedent for a loose dir graduating to a bundle without moving.
  • CC name-collision mechanics (2026-07-10) Plugin identity keys are name@marketplace — collision resolution rides the DECLARING CONTEXT, never the source repo. Within one marketplace duplicates hard-fail validation; across marketplaces coexistence is free; the one unhandled case (two same-named plugins enabled → shared /name: prefix) is undefined behavior — a hole clawker's flat-namespace + declare-time hard error structurally avoids. Marketplace entry name overrides the plugin's own manifest name (docs: "the marketplace entry name is what enabledPlugins keys and /plugin use") — declaration-site rename is a first-class CC pattern.
  • Live-stack seeding is real (2026-07-10) Everything a monitoring extension ships except collector routing is an idempotent REST apply against a running OpenSearch/OSD (templates, pipelines, ISM, saved objects). Collector routing = regen + one-container recreate. Means monitoring consumption does NOT require stack-teardown or a host-pinned enable registry — it can be an at-up-time projection of resolved project config.
  • Collision prior-art deep dive (2026-07-10) CC marketplace layer: name = arbitrary maintainer branding in marketplace.json, no rename/alias flag, keyed by name in extraKnownMarketplaces/known_marketplaces.json; ONLY defined collision error = reserved-name validator (#18329 blocks official/anthropic/claude combos; #46786 reserved list undocumented); non-reserved duplicate marketplace names UNDOCUMENTED (inferred first-registered-wins, open design gap #45266 proposing npm scopes); qualified name@marketplace install broken (#20593 qualifier ignored, first match wins, wrong-marketplace error text); same-source re-add idempotent. codex: duplicate marketplace name from different source = hard error at add (validate_marketplace_name_for_add), no alias anywhere; within-marketplace duplicate plugin names = first-wins + logged warning (not rejection); cross-marketplace = name@marketplace compound ID everywhere (config.toml keys, cache plugins/cache/{marketplace}/{name}/{version}), bare ambiguous install → prompt; plugin.json name must match requested name at install or error. Philosophy: codex solves collisions structurally (compound IDs, per-source cache dirs), hard-errors only at the flat layer (marketplace names); CC left its flat layer undefined and is paying in open bugs.
  • Dead-doc vocabulary resolved (2026-07-10) The former "which contract laws survive" question dissolved on grounding: per-harness overlays = build.harnesses.<name> (live shipped build config, docs/custom-images.mdx) and lineage rendering = the image build chain — both orthogonal to distribution, untouched. Egress floors = harness component content in harness.yaml — integral, untouched. Vendored stacks = dead term for the old stacks-inside-harness hierarchy, subsumed by peers-in-bundle. docs/stack-contract.mdx was already purged from the branch (commit 32707187). No feature ruling was ever needed.
  • Manifest specs researched (CC docs + codex source, 2026-07-10) Claude Code (.claude-plugin/plugin.json) and codex (.codex-plugin/plugin.json) converge: hidden marker dir holds the manifest; fields = name (kebab, = folder), optional semver version (git-SHA fallback), description/author/homepage/repository/license/keywords + component path fields (string|array, ./-relative) with convention-dir auto-discovery fallback; CC single-skill special case (SKILL.md at root = whole dir is one skill); versioned install cache plugins/cache/{marketplace}/{plugin}/{version}/; CC source shapes incl. github{repo,ref,sha}, git-subdir sparse clone, npm — sha pinning native; enable state in scoped settings keyed name@marketplace; marketplace.json = thin {name, source} listing. Refs: code.claude.com/docs/en/plugins-reference.md, plugin-marketplaces.md; codex plugin-json-spec.md.
