# Bug / Feature Tracker

- [ ] **Project `path_rules` updates never propagate to the firewall store after first seed (regression introduced by CP firewall ownership).** `BootstrapServicesPreStart` (`internal/cmd/container/shared/container_start.go:153`) calls `FirewallAddRules` with the project's full `EgressRules()` on every agent container start. `addRulesToStore` (`internal/controlplane/firewall/handler.go:1014`) dedups by `RuleKey = dst:proto:port` — same-key collisions are silently dropped (`if exists known[key] continue` at line 1028). Result: any change to `path_rules`, `path_default`, `action`, or any non-key field for an already-stored domain is invisible — only domains never seen before propagate. Hard-confirmed: `.clawker.yaml` had 5 path_rules under `raw.githubusercontent.com:tls:443`; store file (`.clawkerlocal/.local/share/clawker/firewall/egress-rules.yaml`) had 3 (the snapshot from when the rule first landed in an empty store). `clawker container stop` + `clawker controlplane down` + `clawker run` did NOT re-seed because the store file lives in `FirewallDataSubdir` and persists across CP container teardown. Pre-CP (host-side `internal/firewall/Manager` daemon) the bug existed in code but the user-visible symptom was masked — likely because daemon-restart cycles + different volume topology made fresh seeds more frequent. Post-CP (#255) the store is durable on the CP bind-mount volume and the trailing-mutation invariant in `FirewallRemove` (`internal/controlplane/firewall/CLAUDE.md`) explicitly preserves the store across `firewall down/up`, so stale path_rules are now permanent until manual intervention. Fix options: (A) merge `path_rules` (union by `path`, caller wins on `path_default`/`action`) when `addRulesToStore` sees an existing key — yaml is the project source-of-truth; (B) add `FirewallUpsertRule` RPC and switch the bootstrap caller; (C) in `BootstrapServicesPreStart`, do remove+add per project rule (crude — clobbers any operator-added rule for that key). Recommend (A). Workaround: `clawker firewall remove <domain>` then restart agent container (NOT `controlplane restart` — that doesn't trigger `BootstrapServicesPreStart`); or hand-edit the store file then `clawker firewall reload`.

- [ ] **Replace FNV domain hash with userspace-allocated identity (Cilium pattern).** `internal/controlplane/firewall/ebpf/types.go::DomainHash` derives a 32-bit FNV-1a from the domain string; that hash keys `dns_cache`, `route_map`, and lands on every netlogger record. Theoretically collision-vulnerable. Cilium uses sequential u32 identities allocated by userspace (`pkg/fqdn/namemanager`, `IPCache`); Tetragon doesn't do per-domain BPF enforcement at all. Fix: CP-side identity allocator, BPF maps re-keyed on `identity`, dnsbpf writes identities not hashes, netlogger reverse map collapses to identity→domain direct read. See Serena memory `initiative_route_identity_allocator` for scope.
